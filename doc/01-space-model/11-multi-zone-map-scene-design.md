# 多区域地图场景设计

## 1. 目标

运营中控台地图用于观察一个正在运行的 Robotaxi 运营网络。当前空间目录包含测试运营中的 `Z-001` 和规划中的 `Z-002`，同时保留未来从单区域复制到多区域运营的扩展边界。

地图不是新的业务对象，也不是业务数据副本。它只负责将当前空间对象、运营对象和画像对象投影为可平移、缩放和选择的场景。

## 2. 事实来源

| 地图信息 | 唯一事实来源 |
| --- | --- |
| 区域名称、范围、类型、状态 | Zone 对象 |
| 地点名称、用途、范围、状态 | Place 对象 |
| 服务能力、范围、容量、状态 | ServiceArea 对象 |
| 道路名称、类型、状态和几何 | Road 与 RoadSegment 对象 |
| 运营中心位置和能力 | OpsCenter 对象 |
| Robotaxi 位置、电量和运营状态 | Robotaxi 对象 |
| 需求和供给关键指标 | DemandProfile 对象 |

地图展示注册表只允许定义字段选择、缩放层级和视觉语义，不得保存名称、状态、位置或画像数值。对象或画像更新后，地图从当前运行态重新派生显示，不需要页面同步动作。

## 3. 空间结构

```text
Map M-001（84 × 40）
├── Z-001（col 0-39，测试运营）
├── 区域连接通道（col 39-44，规划中）
└── Z-002（col 44-83，规划中）
```

`Z-001` 的既有 ID、坐标、道路、地点、服务区域和业务关系不得因为地图扩展而改变。`Z-002` 使用真实空间对象建模，但在启用前保持规划状态，其画像保持草稿状态。

## 4. 运营隔离

1. 规划态道路片段不进入路径图。
2. 规划态服务区域不进入投放目的地候选。
3. 规划态 Zone 和草稿画像不进入需求预测、供需平衡和模拟运行候选。
4. 未来启用 `Z-002` 时，应通过区域启用流程变更对象状态，而不是修改地图页面条件。

## 5. 旧数据兼容

旧运行快照通过空间目录合并服务升级：

- 当前代码中的空间目录负责补充新增静态对象和更新已有静态几何；
- 快照中的 Robotaxi、订单、任务、服务区域占用等运行态数据继续保留；
- 页面不负责迁移或拼装空间对象。

## 6. 地图交互

1. 地图铺在完整工作区底层，右侧详情作为上层对象检查器，不使用卡片边框或常驻说明遮挡空间。
2. 标签采用空间锚点、短引线和轻字重文字，并通过反向缩放保持稳定视觉尺寸，不随地图放大而变粗变大。
3. 低缩放显示 Zone 和主要地点；进入细节缩放后隐藏区域标签，改为显示服务区域和道路名称。
4. 鼠标悬停从当前对象和画像读取最多两项关键指标，标题和类型不在指标列表中重复。
5. 点击 Cell 显示唯一选中框；点击任何对象均关闭旧悬浮信息，并通过统一选择入口打开右侧详情。
6. 手机端不依赖悬停，点击对象直接进入详情。
7. 图层说明按需打开，关闭后不占用地图面积。

## 7. 性能边界

- 网格使用一个 SVG Pattern，不为 3360 个 Cell 建立独立交互节点。
- 道路使用 RoadSegment Cell 合并面，服务区域和 Zone 使用合并几何图层；不得用一条中心线替代道路空间事实。
- 拖动期间通过 `requestAnimationFrame` 更新纯视口变换，结束后才提交 React 视口状态。
- 地图移动和缩放不写入运行快照，不触发业务计算或模拟调度。
- Robotaxi 只对存在当前位置的车辆生成轻量投影。

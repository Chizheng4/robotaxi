# Robotaxi 经营分析与统一数据池设计

## 1. 系统定位

经营分析是经营模型的事实观测和反馈环节，连接经营规划、业务执行和下一轮决策。整体经营结构以 `../00-operating-model-overview.md` 为准；本文件只定义指标语义、数据池和分析交付。

```text
经营目标 + 需求画像 + 预测结果 + 供给计划
  → 规划基线
业务单据 + Robotaxi + 行驶 + 成本 + 收入
  → 经营事实
规划基线 + 经营事实
  → 统一经营数据池
  → 目标值 / 预测值 / 实际值 / 差异值 / 达成率 / 偏差原因
```

业务单据生命周期仍是底层事实，模拟运行仍只负责统一时间和调度业务服务。经营分析只消费事实，不修改业务状态，也不为模拟运行建立第二套计算。

## 2. 唯一数据与语义原则

1. 订单、行驶、Robotaxi、成本、收入、规划结果等事实继续由各自服务拥有。
2. 所有经营页面、经营图表和地图经营摘要只能通过统一经营数据池读取经营数据，不在页面拼接或重新计算。
3. 一个经营含义只能有一个稳定指标编号、中文名、单位、公式、时间口径和维度合同。
4. 同义不同名必须合并；同名不同义必须拆分为不同指标。
5. 缺失值是质量事实，不得按 `0` 进入平均值和比率。
6. 字段、指标、单位和枚举中文统一由字段字典服务提供。

## 3. 分层架构

### 3.1 事实层

- 规划事实：BusinessTarget、LongTermDemandForecastResult、SupplyPlan；
- 需求与服务事实：ServiceOrder、Trip、OrderMatchingDecision；
- 资产与供给事实：Robotaxi、ReadinessTask、DeploymentTask、ProductionBatch、DeliveryOrder；
- 行驶事实：RouteExecution、Trip、RoutePlanningRun；
- 财务事实：RevenueRecord、CostRecord；
- 时间事实：业务真实发生时间、统一模拟时间和 SimulationRun。
- 决策事实：各价值流独立拥有的策略配置、策略执行和策略结果。

### 3.2 统一经营数据池服务

统一服务负责：

- 按统计周期、时间口径、Zone、Place、ServiceArea、Robotaxi 等维度筛选事实；
- 选择有效经营目标和不可变预测结果作为规划基线；
- 调用唯一指标定义和计算函数；
- 生成指标观测、质量结果和来源引用；
- 生成规划对比结果和页面分析模型；
- 计算决策过程指标，并向决策中心提供与经营效果指标一致的只读数据；
- 向经营分析、地图摘要和其他前端提供同一份只读数据。

统一数据池不得复制底层事实，也不得让前端直接访问多个事实集合后自行得出经营结论。

### 3.3 指标语义合同

每个指标定义至少包含：

- 指标编号与唯一中文名；
- 业务含义、正式公式和单位；
- 来源对象、来源字段和事实过滤条件；
- 时间口径、统计窗口和支持维度；
- 分子、分母、缺失值和零分母规则；
- 是否越高越好、质量状态和定义版本。

`MetricDefinition` 是底层语义合同，不是要求普通用户单独进入定义页面才能理解指标。分析页面必须直接提供简要含义、周期、单位和质量说明；完整公式、来源和版本按需展开。指标定义页面只服务数据治理、审计和模型维护。

### 3.4 计算记录

计算记录保存统计周期、模型与算法版本、输入范围、成功和受影响指标以及建议动作，用于重算、审计和异常诊断。正常成功时只在分析画布显示数据更新时间；部分成功或失败时才突出问题处理入口。

### 3.5 分析模型

分析结果统一表达：

|字段|含义|
|---|---|
|目标值|经营目标声明的目标|
|预测值|与当前统计周期对齐的需求预测或供给计划值|
|实际值|经营事实计算结果|
|预测差异|实际值减预测值|
|目标差异|实际值减目标值|
|达成率|实际值除目标值，方向相反的指标使用专用解释|
|偏差状态|达标、关注、偏离或数据不足|
|偏差原因|由需求、供给、服务、资产、财务和数据质量驱动项解释|

长期规划按相对经营日对齐：当前经营事实位于预测起点后的第 N 日时，使用预测趋势中相同或最近的周期点比较。期末战略目标单独显示，不伪装成当前应达值。

## 4. 第一阶段指标体系

### 4.1 需求与增长

- 实际日均订单量；
- 同期预测日订单量；
- 期末目标日订单量；
- 需求预测偏差和目标距离；
- 每小时、每日和时段订单趋势。

### 4.2 供给与资产

- 当前有效 Robotaxi；
- 预测所需 Robotaxi；
- Robotaxi 供给缺口；
- 计划生产、实际生产、计划交付和实际交付；
- 履约车辆覆盖率；
- 资产时间利用率。资产时间利用率定义为有效履约时间除以可运营时间，不再使用“有过订单的车辆占全部车辆”冒充利用率。

### 4.3 服务与效率

- 创建、完成、取消和失败订单数；
- 订单履约率、匹配成功率和履约完成率；
- 平均履约距离和平均履约耗时；
- 路径规划成功率；
- 数据缺失的完成订单不进入平均值分母，并形成质量提示。

### 4.4 财务

- 应收、实收、未收收入；
- 运营成本及成本构成；
- 贡献利润和贡献利润率；
- 单均收入、单均运营成本和单均贡献利润；
- 与经营目标中的单均收入、变动成本和最低贡献利润率比较。

生产投入和资产资本性成本不得混入运营贡献利润，除非后续建立明确的折旧或投资回收指标。

## 5. 时间与维度

1. 模拟事实使用统一模拟时间，人工业务事实使用真实业务发生时间；同一观测只能使用一种时间口径。
2. 没有统计时间或无法归属统计周期的事实不得静默进入所有窗口。
3. 第一阶段支持全量、最近经营日、最近七个经营日，并为日、周、月趋势保留统一窗口模型。
4. 全局结果可下钻到 Zone、Place、ServiceArea 和 Robotaxi；规划比较至少按 Zone 对齐。
5. 统计周期必须保存开始、结束、时间口径、来源运行和规划基线版本。

## 6. 前端交付

经营总览、财务表现、服务分析和过程诊断统一使用全宽白色分析画布：

- 不继承对象列表页的常驻右侧详情、状态筛选、搜索栏和最近任务事件；
- 顶部只保留统计周期、数据更新时间和统一“更新经营数据”入口；
- 依次展示规划基线、核心结论、目标/预测/实际差异、趋势、驱动因素和质量提示；
- 指标公式、来源和明细通过画布内按需展开，不挤占主阅读区域；
- 四个页面消费同一数据池，只按管理视角选择不同指标，不各自计算。
- 经营规划和经营分析共用分析画布、摘要、趋势、解释、质量提示和响应式规则，不保留两套移动端实现。
- 手机端按核心结论、趋势、驱动因素、质量提示顺序单列展示；长序列只在图表内部浏览，页面不得整体左右移动。

## 7. 地图与页面上下文

地图摘要通过统一经营数据池和地图对象服务组合，只显示对象名称、类型以及二至三个关键字段。桌面使用经过摘要和点击详情，手机使用点击摘要；摘要必须自适应且不得重复字段。

页面名称和一句话说明由统一页面上下文注册表提供。菜单、页签、页面标题和详情标题必须保持唯一中文命名；分析页可附加统计周期和数据更新时间，普通页面不重复堆叠说明。

## 8. 质量与性能

- 指标计算必须可追溯到来源对象和规划快照；
- 指标定义版本变化不覆盖历史观测；
- 数据池使用一次聚合、多页面复用和 memoized 只读视图，禁止渲染期间反复全量扫描；
- 地图摘要按对象编号索引，悬停不得触发全数据重新计算；
- 模拟运行结束后统一刷新一次，人工业务操作继续由用户触发更新；
- 新架构不得进入模拟逐 Tick 主路径。

# 版本与 Git 规则

## 1. 版本记录

每次稳定版本提交前必须更新：

`VERSION.md`

版本记录必须使用中文，突出：

- 做了什么；
- 为什么重要；
- 影响哪些对象、页面或文档；
- 是否只改文档、只改代码或同时改动。

避免写成冗长流水账。

## 2. 版本号

版本号不得覆盖历史版本。

大版本阶段可使用：

- `v019`
- `v020`

阶段内子版本可使用：

- `v019.1`
- `v019.2`

补丁或连续小迭代可使用：

- `v019.8.1`
- `v019.8.2`

用户明确要求使用某种编号方式时，以用户确认的方式为准。

## 3. 提交前检查

提交前必须确认：

1. `git status` 只包含本轮相关文件；
2. `git diff` 与本轮目标一致；
3. 没有混入用户未确认的无关改动；
4. 必要校验已执行；
5. 前端类改动已做核心路径页面模拟。
6. Major 或 Minor 完成版本已完成对应归档，current 目录没有遗留已完成计划。
7. `VERSION.md` 已写入本轮版本条目，且条目内容与本次提交一致（每个子版本独立一条，不合并）。
8. `scripts/verify-current-iteration-archive.mjs` 已通过，防止已归档版本计划继续留在 current 目录。

## 4. Git 提交

提交信息应简洁表达版本核心，例如：

`v019.8.2 organize iteration rules`

提交后必须创建对应 tag。

对于自动执行计划的最后一个子版本，归档操作必须包含在该子版本提交中。

## 5. 用户修改保护

工作区中出现用户修改时：

- 不得回滚用户修改；
- 不得覆盖用户修改；
- 先判断是否与本轮任务相关；
- 如相关且存在冲突，先说明冲突并等待确认；
- 如无关，只处理本轮文件。

## 6. 不提交的情况

以下情况不应直接提交：

- 用户要求先分析、先确认；
- 文档或代码仍有冲突未确认；
- 前端关键路径无法验证；
- 工作区混入无关改动且无法安全拆分；
- 用户明确要求先不要提交。

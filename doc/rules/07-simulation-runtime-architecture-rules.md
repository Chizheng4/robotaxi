# 模拟运行架构、性能和扩展接入规则

本文件是 Robotaxi 自动化运营模拟运行的长期架构规则。凡涉及模拟运行、时间作业、自动工作流、业务单据生命周期、业务动作自动执行或高速模拟性能的迭代，必须读取并遵守本文件。

## 1. 定位

模拟运行是全系统的统一时间世界和自动执行底座，不是某个业务页面的局部功能。

模拟运行只负责：

1. 维护统一时间；
2. 判断下一个业务触发点；
3. 调度到期业务动作；
4. 汇总运行快照、事件和异常；
5. 调用业务动作服务。

模拟运行不得自己实现业务单据创建、状态变更、路径规划、订单匹配、结算、支付或其他业务闭环逻辑。

## 2. 业务闭环唯一来源

人工点击、真实时间自动化和模拟时间自动化必须调用同一套业务动作服务。

禁止：

- 在模拟循环中直接伪造业务状态；
- 为模拟单独写一套业务状态流转；
- 绕过 `businessActionService`、执行引擎或统一工作流注册表；
- 为某个业务单据单独写特殊模拟逻辑，除非该逻辑已经沉淀为通用调度能力。

业务单据状态时间线属于业务动作服务的生命周期事实，必须在业务动作推动对象状态时同步沉淀。模拟运行不得在结束后用独立补算功能生成或覆盖状态时间线；成本、收入和后续指标体系应消费业务动作已经沉淀的状态时间、模拟创建/完成时间、行驶距离和履约事实。

所有参与展示和闭环操作的业务单据，必须先保持自身服务闭环。模拟运行只能调度业务动作服务，不能把多个单据的状态机合并为一条运行态状态线，也不能把一个单据的状态时间线投影成另一个单据的事实。跨单据联动的正确结构是：源单据调用目标单据服务，目标单据记录自己的状态和事实，源单据接收目标单据返回的业务结果后，用源单据自己的动作和状态继续闭环。

服务订单的 Robotaxi 分配能力必须沉淀在服务订单业务动作服务中。人工分配、真实时间自动化和模拟时间自动化都应复用同一底层匹配服务：

- 人工分配是一次性匹配动作，不启动连续自动分配会话；
- 自动限时分配是服务订单 `WAITING_ROBOTAXI_ASSIGNMENT` 阶段的生命周期能力，不属于模拟运行特有逻辑；
- 自动限时分配应使用统一世界时间推进尝试次数、已耗时、剩余时间、最近失败原因和成功/超时结果；
- 模拟运行只能调度 `ORDER_AUTO_ASSIGNMENT` 时间作业并调用服务订单自动分配动作，不能在模拟循环中自行实现匹配重试或订单取消。

运营行驶记录是 Robotaxi 执行运营任务的通用行驶服务，不是运营投放任务的私有对象。运营投放、运营准入、清洁、充电、维修、故障处理、退役等任务如果需要 Robotaxi 移动，必须通过同一类运营行驶记录服务创建路径规划、行驶推进和到达确认事实。页面、策略和模拟运行不得直接拼装 routeExecution。

新增 Fleet Operation 等基础业务单据时，先完成自身生命周期闭环和服务化动作。模拟运行是上层扩展，未声明模拟接入合同前，不得把新增任务单或其行驶记录加入模拟运行主扫描路径。

## 3. 时间语义

`tick_seconds = 1` 表示统一世界时间的最小精度，不表示所有业务每秒触发。

业务触发必须声明自己的触发周期或到期时间：

- 需求生成可以按 10 分钟等业务周期触发；
- 供给侧准入和投放可以按运营配置时间窗口内的业务周期触发；
- 行驶完成根据路径距离、移动步数和速度计算到期时间；
- 订单等待、Worker 检查、上车、下车、结算、支付等动作根据工作流时效配置计算到期时间。

工作流时效配置是业务单据生命周期自动化推进的耗时合同。业务动作服务写入状态时间线时，必须按 `业务对象类型 + 来源状态 + 动作类型` 解析活动时效规则，并沉淀动作开始时间、状态变更时间、配置耗时和规则编号；成本、收入和指标不得再依赖模拟结束后的独立基础时间补算。

同一模拟时间轴上的多次 SimulationRun 必须区分“计划触发窗口”和“收尾完成事实”：

- `simulation_start_seconds` 是本次运行在统一世界时间上的触发窗口起点；
- `planned_simulation_end_seconds` 是本次运行停止新增需求和供给触发的计划窗口终点；
- `simulation_end_seconds` 是排空既有业务动作后的实际完成时间，只用于记录收尾事实和经营事实发生时间；
- 未重置模拟数据时，下一次 SimulationRun 的触发窗口必须从上一运行的 `planned_simulation_end_seconds` 开始，不能从 `simulation_end_seconds` 开始，否则会跳过上一计划窗口终点到收尾完成之间本应触发的需求或供给周期；
- 重置模拟数据后，新的第一条 SimulationRun 必须从统一世界时间 `0` 秒，即 `Day 1 00:00:00` 开始；
- `current_global_tick` 必须由统一世界秒和 `tick_seconds` 推导，不能作为独立事实时间来源。

## 4. 事件驱动优先

模拟运行应优先面向业务事件推进，而不是无意义逐秒穷举。

短期允许按 1 秒 Tick 兼容推进，但必须遵守：

1. 空 Tick 不得产生大量事件；
2. 空 Tick 不得触发前端持久化和渲染风暴；
3. 业务对象扫描必须逐步收敛到活跃对象集合；
4. 时间作业必须逐步收敛到到期索引或调度队列；
5. 前端展示快照必须节流。

当前运行只能扫描当前 `simulation_run_id` 相关的活跃业务对象。历史运行对象、终态对象和与当前运行无关的业务事实不得进入工作流自动推进扫描；这些历史事实仍可被成本、收入和指标等经营分析模块按统计周期消费。

活跃业务对象必须由统一作用域服务或等价的接入合同判断，不允许在单个业务循环里临时写死。每个参与模拟自动推进的业务对象都应声明主键、状态字段、终态集合和 `simulation_run_id` 归属规则。

时间作业推进必须优先使用候选集合，不得在每个 Tick 无条件重写全部活跃时间作业。候选集合至少包含：当前运行已到期作业、首次进入运行态需要标记的作业、以及 UI 快照点需要刷新展示进度的行驶作业。未到期长作业不得每秒更新 elapsed / remaining / progress，除非该 Tick 是约定的展示快照点。

长期目标是事件驱动跳时：当两个业务事件之间没有可观察状态变化时，模拟内核可直接跳到下一个业务事件时间点，同时保持模拟时间和业务结果准确。

事件驱动跳时只能跳过确定无业务动作的空闲时间段。当前秒是需求触发点、供给触发点、模拟计划结束边界、存在活跃工作流对象、存在到期时间作业或存在首次运行时间作业时，必须执行普通 Tick。跳时只推进统一世界时间游标和 Tick 计数，不得执行、伪造或跳过任何业务动作服务。

计划 Tick 结束只表示停止新需求和供给触发，不表示模拟运行完成。`DRAINING` 阶段必须继续推进当前运行已经产生的业务对象和时间作业；只有当前运行无待执行工作流动作、无活跃业务对象、无未完成时间作业时，SimulationRun 才能进入完成态。

## 5. 前端展示态与运行态分离

高速模拟内核不得把 React state 当作逐秒计算容器。

要求：

- 运行态可以在普通 JS 对象、Map 或专用内核上下文中高速更新；
- React 只接收阶段性快照、业务事件和必要进度；
- Tick 批次内业务集合更新应优先写入运行态缓冲对象，真实执行周期结束后按脏集合批量刷新 React state；不得在高倍速模拟中让每个业务动作都直接触发页面展示态重渲染；
- 事件列表只记录业务事件、异常和检查点，不记录每个 Tick；
- IndexedDB / localStorage 持久化必须节流或批量提交；
- 调试日志必须可关闭并设置上限。
- 成功终态诊断数据可以在运行完成后压缩或清理；失败、取消、异常和当前运行未完成数据必须保留，保证排查能力不丢失。

## 6. 新业务接入合同

新增业务单据、状态节点、动作、策略或时间作业时，必须声明：

1. 业务对象类型；
2. 状态机和终态；
3. 可执行动作；
4. 动作处理器；
5. 动作耗时来源；
6. 是否受工作时间或运营时间约束；
7. 是否创建时间作业；
8. 是否参与模拟触发；
9. 是否参与活跃对象扫描；
10. 产生或更新哪些模拟时间字段；
11. 字段字典和中文展示。

没有接入合同的新增业务，不得进入模拟自动化主路径。

当前阶段参与模拟自动推进的 `routeExecution` 只包括已声明的投放类运营行驶记录。其他任务单创建的运营行驶记录可以存在于同一集合中，但必须通过任务类型或接入合同与模拟扫描隔离，直到对应业务完成模拟接入设计和验证。

新增运行态集合、策略执行记录、调度记录或业务结果集合时，还必须同步接入：

1. `loadRuntimeSnapshot` 的默认 fallback；
2. IndexedDB/localStorage 恢复兼容；
3. `saveRuntimeSnapshot` 的持久化结构；
4. 序列号恢复；
5. `rowsByPage`、`pageObjectType`、`idFieldByType`、`statusFieldByPage` 等页面数据合同；
6. 真实浏览器加载验证，确认旧快照、空快照和新增页面不会导致白屏。

所有被上述初始化、恢复或首屏页面使用的服务模块，必须在 `bootstrap` 的 `Promise.all` 中等待完成。禁止把这类服务放到独立 `import().then(...)` 分支后再让 React 先渲染。

## 7. 性能验收

模拟性能迭代必须验证：

1. 不记录 Tick 级事件洪流；
2. 运行时事件数量有上限或按业务事件增长；
3. 前端持久化有节流；
4. 调试日志有上限；
5. 时间作业推进不会无意义重写全部 state；
6. 业务结果仍通过原业务动作服务产生；
7. 页面入口可正常加载；
8. 提交前检查通过。

## 8. 规则维护

如未来新增模拟架构能力，应优先扩展本文件，而不是在迭代文档、页面代码或单个业务服务中形成第二套规则。

# 模拟运行架构、性能和扩展接入规则

本文件是 Robotaxi 自动化运营模拟运行的长期架构规则。凡涉及模拟运行、时间作业、自动工作流、业务单据生命周期、业务动作自动执行或高速模拟性能的迭代，必须读取并遵守本文件。

## 1. 定位

模拟运行是全系统的统一时间世界和自动执行底座，不是某个业务页面的局部功能。

模拟运行只负责：

1. 维护统一时间；
2. 判断下一个业务触发点；
3. 调度到期业务动作；
4. 汇总运行快照、事件和异常；
5. 调用业务动作服务。

模拟运行不得自己实现业务单据创建、状态变更、路径规划、订单匹配、结算、支付或其他业务闭环逻辑。

## 2. 业务闭环唯一来源

人工点击、真实时间自动化和模拟时间自动化必须调用同一套业务动作服务。

禁止：

- 在模拟循环中直接伪造业务状态；
- 为模拟单独写一套业务状态流转；
- 绕过 `businessActionService`、执行引擎或统一工作流注册表；
- 为某个业务单据单独写特殊模拟逻辑，除非该逻辑已经沉淀为通用调度能力。

## 3. 时间语义

`tick_seconds = 1` 表示统一世界时间的最小精度，不表示所有业务每秒触发。

业务触发必须声明自己的触发周期或到期时间：

- 需求生成可以按 10 分钟等业务周期触发；
- 供给侧准入和投放可以按运营配置时间窗口内的业务周期触发；
- 行驶完成根据路径距离、移动步数和速度计算到期时间；
- 订单等待、Worker 检查、上车、下车、结算、支付等动作根据工作流时效配置计算到期时间。

## 4. 事件驱动优先

模拟运行应优先面向业务事件推进，而不是无意义逐秒穷举。

短期允许按 1 秒 Tick 兼容推进，但必须遵守：

1. 空 Tick 不得产生大量事件；
2. 空 Tick 不得触发前端持久化和渲染风暴；
3. 业务对象扫描必须逐步收敛到活跃对象集合；
4. 时间作业必须逐步收敛到到期索引或调度队列；
5. 前端展示快照必须节流。

长期目标是事件驱动跳时：当两个业务事件之间没有可观察状态变化时，模拟内核可直接跳到下一个业务事件时间点，同时保持模拟时间和业务结果准确。

## 5. 前端展示态与运行态分离

高速模拟内核不得把 React state 当作逐秒计算容器。

要求：

- 运行态可以在普通 JS 对象、Map 或专用内核上下文中高速更新；
- React 只接收阶段性快照、业务事件和必要进度；
- 事件列表只记录业务事件、异常和检查点，不记录每个 Tick；
- IndexedDB / localStorage 持久化必须节流或批量提交；
- 调试日志必须可关闭并设置上限。

## 6. 新业务接入合同

新增业务单据、状态节点、动作、策略或时间作业时，必须声明：

1. 业务对象类型；
2. 状态机和终态；
3. 可执行动作；
4. 动作处理器；
5. 动作耗时来源；
6. 是否受工作时间或运营时间约束；
7. 是否创建时间作业；
8. 是否参与模拟触发；
9. 是否参与活跃对象扫描；
10. 产生或更新哪些模拟时间字段；
11. 字段字典和中文展示。

没有接入合同的新增业务，不得进入模拟自动化主路径。

## 7. 性能验收

模拟性能迭代必须验证：

1. 不记录 Tick 级事件洪流；
2. 运行时事件数量有上限或按业务事件增长；
3. 前端持久化有节流；
4. 调试日志有上限；
5. 时间作业推进不会无意义重写全部 state；
6. 业务结果仍通过原业务动作服务产生；
7. 页面入口可正常加载；
8. 提交前检查通过。

## 8. 规则维护

如未来新增模拟架构能力，应优先扩展本文件，而不是在迭代文档、页面代码或单个业务服务中形成第二套规则。

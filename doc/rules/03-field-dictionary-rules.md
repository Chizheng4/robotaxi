# 字段字典规则

## 1. 统一字段字典

字段字典有两个文件，必须同时维护：

- `doc/rules/field-dictionary.md` — 文档版，供人阅读
- `src/domain/fieldDictionary.js` — 前端代码版，供程序使用

代码版分为两个对象：
  - `fieldDictionary`：字段英文名 → 中文显示名（表格列头、详情字段名）
  - `valueDictionary`：枚举值 → 中文显示名（状态标签、筛选选项）

所有业务对象、单据、策略、执行记录、状态、枚举、前端展示字段必须同时在这两个文件中更新。

业务目录下不再维护独立字段字典，只保留迁移说明或引用入口。

## 2. 字段定义原则

同一业务含义只能有一个稳定英文字段名。

字段字典至少包含：

- 英文字段名；
- 中文显示名；
- 字段性质；
- 字段含义；
- 必要时包含枚举取值和中文值。

承担经营目标、指标或跨页面复用的关键字段，还必须在代码语义注册中声明所属对象、类型、单位、业务定义、指标概念和来源关系。页面不得硬编码这些字段的中文名称。

只有业务含义、粒度、单位和计算口径完全一致时才复用同一字段。含义发生变化时必须新增字段并声明旧字段兼容关系，禁止静默改变旧字段语义。

## 3. 新增或修改字段

以下变化必须同步字段字典：

- 新增业务对象；
- 新增单据；
- 新增策略或执行记录；
- 新增字段；
- 字段含义变化；
- 状态或枚举变化；
- 前端显示名称变化。
- **对象内部的嵌套子对象字段（如 config、snapshot、profile 内的字段）同样必须录入字段字典。** 仅录入顶层字段是不够的，详情面板渲染嵌套对象时会逐字段查找中文标签。

## 4. 中文展示规则

前端表格、详情栏、筛选项、状态分类、操作上下文必须优先显示中文。

不得向运营人员直接暴露英文枚举值或内部字段名。

前端展示字段名、对象名和枚举值必须通过统一字段展示服务获取；字段字典缺失不得依赖静默回退长期存在。新增或修改字段后，必须让字段展示合同检查通过。

枚举优先使用字段专属映射，避免相同英文值在不同业务上下文中被错误翻译；仅无字段上下文时才使用全局映射。

状态枚举必须优先表达所属单据自己的业务含义。不得长期使用一个通用状态值再依赖对象类型、任务类型或页面上下文组合出不同中文含义；如果同一枚举值在清洁、充电、维修、行驶等单据中需要不同展示或不同业务解释，必须拆成单据自身明确枚举，并同步字段字典文档和代码。

如发现前端出现英文代码值，优先检查：

1. 字段字典是否缺失；
2. 前端字段映射是否缺失；
3. 枚举中文值是否缺失；
4. 当前对象是否使用了兼容旧字段。

## 5. 兼容字段

兼容字段只用于旧数据或旧前端展示过渡。

新增代码不得优先依赖兼容字段。

发现兼容字段与新字段语义重复时，应在本轮分析中指出，并按用户确认逐步收敛。

## 6. 位置结构化表达

涉及位置的对象必须优先使用统一结构化表达。

Robotaxi、任务、行驶记录、服务订单等对象展示位置时，应尽量解释：

- Cell；
- ServiceArea；
- RoadSegment；
- Road；
- Place；
- Route；
- 当前业务上下文。

详情区默认展示摘要，完整上下文可折叠。

## 7. 每轮编码结束时的强制检查

每轮编码结束后，准备告知用户「完成」之前，必须检查：

1. 本轮新增或修改了哪些业务对象、字段、状态、枚举；
2. `doc/rules/field-dictionary.md` 是否已包含上述所有内容；
3. 枚举值字典是否已包含所有新增枚举的中文映射。
4. `scripts/verify-field-display-contract.mjs` 是否覆盖本轮新增字段，并已通过检查。

任何未同步的字段或枚举，必须在告知用户完成前补充。

此要求适用于所有编码任务，不分 Major 或 Minor。

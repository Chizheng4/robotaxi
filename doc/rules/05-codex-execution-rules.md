# Codex 执行规则

## 1. 默认工作方式

Codex 每轮任务先确认：

- 当前 Git 状态；
- 当前版本；
- 用户最新请求；
- 是否只分析、只整理文档、还是可以编码；
- 相关规则和业务文档。

用户明确要求“先不要编码”时，只做分析或文档整理。

用户明确要求“开始迭代”时，才进入代码实现。

## 2. 分析输出

方案或文档更新后的分析默认包含：

1. 本次文档变更；
2. 最新业务模型理解；
3. 当前代码与文档差异；
4. 字段、状态、命名和对象职责冲突；
5. 风险与疑问；
6. 建议迭代范围；
7. 计划读取文件；
8. 计划修改文件；
9. 需要用户确认的问题。

## 3. 编码要求

编码必须遵守：

- 最小可运行；
- 不重写无关文件；
- 不提前实现未来对象；
- 对象、初始化、校验、前端展示分层清晰；
- 单据操作影响对象状态时，同步更新相关对象；
- 代码、字段字典、文档和前端展示保持一致。

## 4. 验证要求

编码后必须执行与本轮范围匹配的验证。

前端交互类改动必须做页面模拟测试，确认核心路径可用。

如果验证失败，先修复再提交。

如果验证受环境限制无法完成，必须在最终说明中明确说明。

## 5. 暂停确认条件

出现以下情况必须暂停并向用户确认：

- 文档之间存在业务冲突；
- 当前代码实现与用户最新方案方向冲突；
- 需要扩大本轮范围；
- 会影响历史版本或旧版 demo；
- 字段命名存在多种合理选择；
- 自动计划无法继续闭环；
- 前端验证发现核心路径不可用且修复会扩大范围。

## 6. 禁止行为

禁止：

- 未读取规则即执行；
- 以「任务与前端/字段无关」为理由跳过 AGENTS.md 规定的强制必读文件；
- 编码结束后跳过 AGENTS.md 规定的完成自检清单；
- 未确认就编码用户要求先确认的任务；
- 修改历史归档；
- 删除历史版本；
- 覆盖用户修改；
- 提交无关文件；
- 为了当前页面便利而写死业务关系；
- 直接向前端暴露英文枚举和内部字段名；
- 把策略服务设计成主动改变调用方状态，除非业务文档明确要求。

## 7. 回复规则

最终回复应简洁说明：

- 完成了什么；
- 改动了哪些关键文件；
- 是否验证；
- 是否存在未完成或需要确认的问题。

不要用大量流水账替代工程结论。

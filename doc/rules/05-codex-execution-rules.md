# Codex 执行规则

## 1. 默认工作方式

Codex 每轮任务先确认：

- 当前 Git 状态；
- 当前版本；
- 用户最新请求；
- 是否只分析、只整理文档、还是可以编码；
- 相关规则和业务文档。

用户明确要求“先不要编码”时，只做分析或文档整理。

用户明确要求“开始迭代”时，才进入代码实现。

## 2. 分析输出

方案或文档更新后的分析默认包含：

1. 本次文档变更；
2. 最新业务模型理解；
3. 当前代码与文档差异；
4. 字段、状态、命名和对象职责冲突；
5. 风险与疑问；
6. 建议迭代范围；
7. 计划读取文件；
8. 计划修改文件；
9. 需要用户确认的问题。

## 3. 编码要求

编码必须遵守：

- 最小可运行；
- 不重写无关文件；
- 不提前实现未来对象；
- 对象、初始化、校验、前端展示分层清晰；
- 单据操作影响对象状态时，同步更新相关对象；
- 代码、字段字典、文档和前端展示保持一致。
- 人工操作、自动化操作和策略触发必须优先复用同一业务服务；禁止为页面按钮、模拟运行或策略执行各写一套对象创建逻辑。
- 策略执行记录必须保存本次使用的配置快照和关键输入输出，保证策略配置后续变化后仍可复盘历史执行结果。
- 运营行驶记录是 Robotaxi 执行运营任务的通用底层服务。新增任务单需要车辆移动时，只能调用服务化行驶能力创建、规划、推进和确认到达，禁止在页面层拼装 routeExecution。
- 新增业务任务单默认只是业务基础闭环，不自动进入模拟运行；只有完成模拟接入合同后，模拟运行才可以调度它。

## 4. 验证要求

编码后必须执行与本轮范围匹配的验证。

前端交互类改动必须做页面模拟测试，确认核心路径可用。

修改 `src/main.jsx` 的 bootstrap、动态 import、运行态初始化、持久化恢复、页面入口或启动脚本时，必须通过真实浏览器加载门禁，不能只依赖 bundle 语法检查。标准门禁为：

- 已有服务运行时执行 `node scripts/verify-browser-load.mjs`；
- 或执行 `./start-robotaxi.command`，由启动脚本完成编译、服务就绪和真实浏览器渲染检查。

真实浏览器检查必须确认 `#app` 存在、工作台主体 `.workbench` 已渲染，并捕获 `ReferenceError`、`TypeError`、`SyntaxError`、React 运行时异常等白屏风险。

如果验证失败，先修复再提交。

如果验证受环境限制无法完成，必须在最终说明中明确说明。

## 4.1 自动执行计划收口

用户要求“按照版本计划直接执行”“自动执行完迭代”“不要停下来”时，Codex 必须把当前计划当作连续交付队列处理，不能只完成其中一段后自行停止。

自动执行计划在最终回复前必须完成以下收口检查：

1. 当前迭代文档中所有标记为“待执行”的本轮计划项，要么已完成并改为“已完成”，要么因规则中的暂停条件明确阻塞并说明原因；
2. 当前迭代文档中最新“已完成”子版本必须同步写入 `VERSION.md`；
3. 稳定版本必须完成提交并打对应 tag，tag 版本号必须与最新完成子版本一致；
4. 大版本计划全部完成后，计划文件必须从 `doc/common/current-iteration/major/` 移入 `doc/common/iteration-history/major/`；不能因为该文件仍被 `major-current-iteration.md` 声明为当前计划而继续留在 current 目录；
5. `scripts/verify-current-iteration-archive.mjs` 和 `scripts/verify-automated-plan-closure.mjs` 必须通过；
6. 最终回复前必须用 `git status --short` 和 `git tag --points-at HEAD` 确认没有漏提交、漏打 tag。

禁止在计划文档已经标记后续版本完成、但 `VERSION.md`、提交或 tag 仍停留在更早版本时声称完成。

## 5. 暂停确认条件

出现以下情况必须暂停并向用户确认：

- 文档之间存在业务冲突；
- 当前代码实现与用户最新方案方向冲突；
- 需要扩大本轮范围；
- 会影响历史版本或旧版 demo；
- 字段命名存在多种合理选择；
- 自动计划无法继续闭环；
- 前端验证发现核心路径不可用且修复会扩大范围。

## 6. 禁止行为

禁止：

- 未读取规则即执行；
- 以「任务与前端/字段无关」为理由跳过 AGENTS.md 规定的强制必读文件；
- 编码结束后跳过 AGENTS.md 规定的完成自检清单；
- 未确认就编码用户要求先确认的任务；
- 修改历史归档；
- 删除历史版本；
- 覆盖用户修改；
- 提交无关文件；
- 为了当前页面便利而写死业务关系；
- 为了当前按钮能用而在 `src/main.jsx` 拼装业务对象、状态流转、运营行驶记录或策略结果；
- 直接向前端暴露英文枚举和内部字段名；
- 把策略服务设计成主动改变调用方状态，除非业务文档明确要求。
- 将首屏渲染依赖的服务模块放在 `import().then(...)` 这类不等待完成的异步分支中。所有首屏、运行态初始化、fallback、恢复、字段展示和页面配置依赖的服务，必须纳入 `bootstrap` 的统一等待链，确保 React 渲染前已经可用。

## 7. 回复规则

最终回复应简洁说明：

- 完成了什么；
- 改动了哪些关键文件；
- 是否验证；
- 是否存在未完成或需要确认的问题。

不要用大量流水账替代工程结论。

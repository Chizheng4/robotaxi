# Robotaxi 前端设计系统

## 1. 目的与适用范围

本文件是 Robotaxi 运营平台前端视觉和交互一致性的长期标准，适用于 Codex、其他 AI 模型和人工开发者。

凡涉及新增页面、模块、表格、筛选、按钮、详情、地图、事件区、状态展示或样式调整，都必须先读取并沿用本文件。目标是让后续功能像调用固定标准一样自然接入，而不是每次重新决定视觉风格。

本文件与 `04-frontend-ux-rules.md` 的分工：

- `04` 定义 B 端业务页面必须具备的结构和操作闭环；
- `07` 定义这些结构如何保持统一、克制、优雅和可持续扩展；
- 两者同时生效，不能择一执行。

## 2. 产品体验原则

### 2.1 运营效率优先

界面首先服务运营人员理解对象、状态、关系和下一步动作。视觉优化必须降低认知成本，不得用装饰、营销式构图或大面积留白占用业务空间。

### 2.2 安静而清晰

整体气质参考 Codex 类工作界面的克制感：中性表面、近黑文字、细边界、少量强调色、自然留白。复杂业务仍应显得稳定、有秩序、不费力。

### 2.3 层级来自结构

层级优先通过对齐、间距、字号、字重和分组建立。使用顺序：

1. 间距与对齐；
2. 字体层级；
3. 细分隔线；
4. 轻微表面色；
5. 必要边框；
6. 最后才使用阴影。

禁止用卡片堆叠、粗边框、强渐变或大色块代替信息架构。

### 2.4 一致胜过局部惊艳

新模块必须看起来属于同一个产品。除地图语义色和业务状态色外，不得引入新的主色、字体体系、圆角体系或阴影体系。

### 2.5 交互必须可预期

相同动作在不同页面使用相同位置、文案、尺寸和反馈。全局操作、单行操作、状态分类、查询重置、详情联动不得因页面不同而改变基本行为。

## 3. 设计令牌

设计令牌以 `src/styles.css` 的 `:root` 为运行时唯一来源。新增样式优先使用已有变量，不直接散落硬编码颜色。

### 3.1 颜色

|令牌|用途|
|---|---|
|`--bg`|应用背景和弱分区背景|
|`--surface`|主内容、侧栏、详情等基础表面|
|`--surface-soft`|表头、弱提示和轻量分组|
|`--surface-muted`|悬停、次级选中和弱交互反馈|
|`--line`|主要结构分隔线|
|`--line-strong`|输入框、按钮等控件边界|
|`--line-soft`|表格行和低层级分隔线|
|`--text`|主要文字|
|`--muted`|标签、说明和辅助文字|
|`--accent`|当前选中、主操作和关键链接|
|`--accent-soft`|选中行、当前状态和轻强调背景|
|`--success`|成功、可用、运行正常|
|`--warning`|待处理、暂停、注意|
|`--danger`|失败、阻塞、不可用|
|`--info`|进行中、移动、处理中|

规则：

- 主色只用于选择、主操作和关键状态，不铺满大面积背景；
- 状态必须同时有文字，不得只靠颜色表达；
- 新地图语义色必须与现有地点、服务区、运营中心、路线和道路节点颜色协调；
- 禁止新增紫蓝渐变、装饰光斑和无业务含义的高饱和颜色。

### 3.2 字体

- 字体栈统一使用 `Inter`、`SF Pro Text`、`PingFang SC`、`Microsoft YaHei` 和系统无衬线回退；
- `12px`：表格、详情、筛选、状态、说明；
- `13px`：菜单、按钮、重要标签；
- `14px`：模块标题和全局关键标题；
- `16px`：产品品牌上限；
- 字间距固定为 `0`，不使用负字间距；
- 数字表格使用等宽数字特性；
- 通过字重和间距建立层级，不使用超大标题。

### 3.3 间距与尺寸

基础间距序列：`4 / 6 / 8 / 10 / 12 / 14 / 16 / 20 / 24px`。

标准尺寸：

- 顶部全局栏：`52px`；
- 工作台页签栏：`38px`；
- 主导航项：`36px`；
- 标准按钮和状态分类：`28px`；
- 紧凑表格行：`36px`；
- 详情标题栏：`44px`；
- 详情栏标准宽度：桌面 `328px`，窄桌面 `292px`；
- 折叠详情轨道：`40px`；
- 圆角：普通控件 `6px`，不得超过 `8px`，除圆形状态点外不使用胶囊化外观。

## 4. 标准页面骨架

### 4.1 全局工作台

从左到右、从上到下固定为：

1. 左侧品牌与层级导航；
2. 顶部当前模块信息与全局摘要；
3. 单行工作台页签；
4. 中间业务内容；
5. 右侧上下文详情；
6. 必要时在业务内容底部放事件、日志或辅助信息。

禁止在工作台内新增第二套全局导航或重复的大页面标题。

### 4.2 列表页模板

标准顺序不可随意调整：

1. 状态分类 `.status-segment-bar`；
2. 搜索过滤 `.list-filter-bar`；
3. 全局操作 `.list-action-bar`（无全局操作时省略）；
4. 表格 `.record-table-section`；
5. 事件区 `.event-log-section`（按业务需要）；
6. 底部信息 `.module-footer`；
7. 右侧详情 `.detail-rail`。

新增列表模块必须复用 `RecordTable`、统一页面状态结构和以上类名，不复制一份相似页面自行维护。

### 4.3 地图页模板

- 地图是运营中控台的主要画布，使用 `.map-stage` 与 `.zone-canvas-new`；
- 地图工具固定使用 `.map-floating-actions`；
- 图例固定使用 `.legend-new`；
- 详情继续使用统一右侧详情栏；
- 工具不得遮挡关键地图对象，图例不得挤压详情栏；
- 地图语义优先于装饰，不使用真实地图风格伪装模拟网格。

### 4.4 详情模板

- 标题栏统一 `.panel-title`；
- 简单对象使用单列 `Descriptions`；
- 字段较多的对象使用 `Tabs + Descriptions`；
- 字段标签左、字段值右，标签使用弱色，值使用主文字色；
- 状态使用统一状态点与中文文案；
- 详情为空时使用统一空状态，不展示无关地图详情；
- 内容较长时详情内部滚动，不推动全局布局。

## 5. 组件规范

### 5.1 按钮

- 主按钮每个操作组原则上只有一个；
- 次要操作使用默认按钮；
- 行操作复用 `RowActionButton` 和 `RowActionGroup`；
- 图标按钮有标准图标时优先使用图标并提供 tooltip/`title`；
- 危险操作必须使用危险状态并保留确认；
- 禁止同一动作在不同页面使用不同文案。

### 5.2 状态

- 状态值复用 `StatusValue`；
- 成功、警告、危险、进行中和中性分别使用统一语义色；
- 状态分类的当前项使用浅强调背景，不使用大面积实心主色；
- 英文枚举必须通过 `valueDictionary` 显示中文。

### 5.3 筛选

- 必须包含查询与重置；
- 标签、输入框、选择器和按钮保持单行紧凑排列；
- 窄桌面允许减少非关键摘要，但不得隐藏查询、重置或主要筛选字段；
- 筛选结果必须在底部信息区可解释。

### 5.4 表格

- 表头、行高、内边距和分隔线统一；
- 当前行使用 `--accent-soft`，悬停行使用 `--surface-muted`；
- 数字对齐稳定，文本溢出省略，完整内容由详情承接；
- 关键识别字段放前，操作列固定在最右；
- 不为不同模块单独设计另一套表格皮肤。

### 5.5 页签与详情

- 工作台页签只表达已打开页面，不做卡片或胶囊；
- 当前页签使用文字主色和底部细线；
- 详情 Tab 使用同一强调色和紧凑高度；
- 页签溢出横向滚动，不换行。

## 6. 交互状态完整性

新增或修改控件必须覆盖：

- 默认；
- 悬停；
- 聚焦；
- 选中 / 激活；
- 禁用；
- 加载（如存在异步）；
- 空数据；
- 错误（如业务支持）；
- 折叠与展开（如组件支持）。

键盘聚焦必须可见，但不得把整个导航容器画成粗蓝框；聚焦样式应落到具体可操作项。

## 7. 新模块接入协议

任何模型或开发者新增前端模块时按以下顺序执行：

1. 确认模块属于地图页、标准列表页还是现有特殊工作台；
2. 搜索并复用最接近的现有页面、组件和 class；
3. 使用现有设计令牌，不先创建新颜色或尺寸；
4. 接入统一工作台状态、状态分类、查询重置、分页、选中记录和详情；
5. 字段和枚举使用统一字段字典；
6. 仅当现有标准无法表达通用需求时才扩展设计系统；
7. 扩展必须先写入本文件和 `src/styles.css` 令牌，再被业务页面使用；
8. 在 1280 x 720 与 1440 x 900 进行真实页面验证；
9. 完成中文、布局、交互状态和控制台错误自检。

## 8. AI 模型执行约束

为确保更换模型后仍保持一致，执行前端任务的模型必须：

1. 完整读取 `AGENTS.md`、`04-frontend-ux-rules.md`、本文件和 `fieldDictionary.js`；
2. 先查看现有页面和 `src/styles.css`，不得仅凭提示词想象风格；
3. 明确列出复用的页面模板、组件和设计令牌；
4. 不因模型能力、上下文长度或使用量不足跳过本规则；
5. 无法完成真实页面验证时不得声称完成；
6. 发现标准冲突时暂停说明，不自行创造第三套规则；
7. 提交前通过项目检查，并在版本记录中说明是否扩展了设计系统。

## 9. 禁止事项

- 单页硬编码新主色、字体、圆角或阴影；
- 复制已有组件后仅改样式形成分叉；
- 卡片套卡片、页面区块悬浮卡片化；
- 大标题、营销 Hero、装饰渐变、光斑和无业务含义插画；
- 为视觉整齐隐藏必要业务字段或操作；
- 用颜色代替文字状态；
- 直接展示英文枚举或内部字段名；
- 修改页面后不检查详情、空状态、事件区和窄桌面；
- 为当前页面便利写死对象关系或状态逻辑。

## 10. 完成验收清单

- [ ] 页面骨架是否来自标准模板？
- [ ] 是否复用现有设计令牌和组件？
- [ ] 字号、间距、圆角、边界和状态色是否一致？
- [ ] 状态分类、查询重置、操作、表格、详情和底部信息是否闭环？
- [ ] 所有业务字段和枚举是否中文显示？
- [ ] 默认、悬停、聚焦、选中、禁用、空状态是否完整？
- [ ] 1280 x 720 和 1440 x 900 是否无重叠、遮挡和不可操作区域？
- [ ] 地图、表格、事件区和详情边界是否对齐？
- [ ] 是否无新增控制台错误？
- [ ] 是否避免一次性样式和组件分叉？

# 文档差异驱动迭代规则

## 1. 目的

本文档用于固化“用户更新方案文档，Codex 通过 Git diff 识别差异，再进入确认和执行”的协作方式。

目标是减少重复读取、重复分析和重复沟通，让长期迭代更高效，同时保持工程严谨。

## 2. 适用场景

适用于用户先修改方案文档、字段定义、当前迭代文件或大版本计划，然后要求 Codex：

- 识别差异；
- 分析冲突；
- 输出疑问；
- 形成执行计划；
- 经用户确认后自动执行。

不适用于用户直接要求修复明确代码 bug 或执行简单命令。

## 3. 固定口令

用户可以使用以下简短口令触发固定流程：

| 口令            | Codex 默认动作                         |
| ------------- | ---------------------------------- |
| `按文档差异进入迭代分析` | 只做 diff 分析，不编码                     |
| `确认，生成执行计划`   | 把确认后的范围写入 current-iteration        |
| `按执行计划开始自动执行` | 按当前计划连续执行                          |
| `暂停，先提交当前版本`  | 停止继续编码，整理版本记录并提交                   |
| `继续执行当前计划`    | 从执行指针或未完成子版本继续                     |
| `查看当前执行进度`    | 读取 current-iteration 和 Git 状态后汇报进度 |

## 4. 阶段一：用户更新文档

用户可以只修改文档，不需要在对话中重复描述完整方案。

Codex 收到分析请求后，默认只读取：

1. `git status --short`
2. `git diff --stat`
3. `git diff --name-only`
4. 被修改文件的 diff
5. `doc/iteration-rules.md`
6. `doc/rules/00-rule-index.md`
7. `doc/rules/06-document-driven-iteration-rules.md`
8. 必要时读取相关字段字典和少量业务文档

默认不全量扫描项目。

## 5. 阶段二：标准差异分析

Codex 输出差异分析时，默认使用以下结构：

1. 当前版本和工作区状态；
2. 本次变更文件；
3. 文档表达的核心业务变化；
4. 字段变化；
5. 状态变化；
6. 菜单 / 前端变化；
7. 对象职责变化；
8. 当前代码与文档差异；
9. 冲突、矛盾、疑问；
10. 建议拆分版本；
11. 计划读取文件；
12. 计划修改文件；
13. 需要用户确认的问题。

此阶段禁止编码。

## 6. 阶段三：生成执行计划

用户确认差异分析后，Codex 根据范围判断使用 major 或 minor。

使用 major 的情况：

- 跨多个核心对象；
- 涉及状态机重构；
- 涉及多文档重构；
- 涉及菜单结构和代码结构同时变化；
- 需要连续多个子版本执行。

使用 minor 的情况：

- 单个对象字段调整；
- 单个页面体验优化；
- 单个状态显示或按钮文案修正；
- 小范围文档补充。

执行计划写入：

- major：`doc/common/current-iteration/major/`
- minor：`doc/common/current-iteration/minor/minor-current-iteration.md`

major 和 minor 不得混用。

## 7. 阶段四：自动执行

用户确认“按执行计划开始自动执行”后，Codex 默认按计划推进。

每个子版本固定执行：

1. 读取当前执行计划和当前子版本范围；
2. 读取本子版本必读文档；
3. 更新业务文档和字段字典；
4. 编码实现；
5. 生成前端 bundle；
6. 运行提交前检查；
7. 做浏览器核心路径验证；
8. 更新 `VERSION.md`；
9. 提交并打 tag；
10. 更新执行计划状态或指针；
11. 进入下一个子版本。

如果用户明确要求先只做文档，则不进入编码阶段。

## 8. 暂停条件

出现以下情况必须暂停确认：

- 字段含义冲突；
- 状态机无法闭环；
- 对象职责不清；
- 需要新增计划外对象；
- 需要扩大当前子版本范围；
- 文档与当前代码严重不一致；
- 浏览器验证失败且修复会扩大范围；
- 用户在执行过程中更新了同一范围文档；
- 需要执行真实支付、真实地图、真实外部系统或其他未确认能力。

## 9. 节省使用量原则

Codex 应优先使用 Git diff 确定阅读范围。

不应在每次文档差异分析时重复读取全量业务文档。

只有当 diff 指向某个对象、字段、状态或链路时，才读取对应文件。

如果当前计划已经明确子版本范围，执行时只读取该子版本必读文件和直接相关代码。

## 10. 交付要求

每次自动执行完成后，Codex 应简洁说明：

- 完成的子版本；
- 修改的核心文件；
- 验证结果；
- 提交号和 tag；
- 下一步执行指针。

如果只完成文档计划，不提交版本，则必须明确说明当前工作区状态。

# 前端体验与设计系统规则

本文件是 Robotaxi 运营平台前端结构、视觉、交互和扩展方式的唯一长期标准，适用于 Codex、其他 AI 模型和人工开发者。具体任务是否必须读取本文件，只由根目录 `AGENTS.md` 按任务类型判断，其他文档不得建立第二套读取条件。

## 1. 产品原则

- 前端是 B 端运营工作台，不是营销页；首要目标是帮助运营人员快速理解对象、状态、关系和下一步动作。
- 视觉设计者必须以视觉艺术家的审美标准审视每个界面。精致和轻是第一视觉感受，来自准确比例、克制字重、细边界、稳定对齐、呼吸感和细微状态差异，不来自粗大文字、厚重色块或装饰堆叠。
- 整体气质保持安静、清晰、克制：中性表面、清晰文字、细边界、少量强调色和自然留白。
- 业务内容、表格、地图和详情获得最大可用面积；导航、标题、说明和装饰保持紧凑。
- 层级依次通过对齐与缩进、间距、字号与字重、细分隔线和轻表面色建立，阴影只用于确有空间层级的浮层。
- 全站一致性优先于单页新奇。相同动作在不同页面使用相同位置、文案、尺寸和反馈。

## 2. 设计令牌

`src/styles.css` 的 `:root` 是运行时令牌唯一来源。新增样式优先复用变量，不在业务页面散落新颜色、字号、圆角和阴影。

### 2.1 颜色

|令牌|用途|
|---|---|
|`--bg`|应用背景和弱分区背景|
|`--surface`|主内容、侧栏、详情基础表面|
|`--surface-soft`|表头、弱提示和轻量分组|
|`--surface-muted`|悬停和次级交互反馈|
|`--line` / `--line-strong` / `--line-soft`|结构、控件和表格分隔|
|`--text` / `--muted`|主要文字和辅助文字|
|`--accent` / `--accent-soft`|主操作、当前选中和轻强调背景|
|`--success` / `--warning` / `--danger` / `--info`|统一业务语义状态|

- 主色只用于选择、主操作和关键状态，不铺满大面积背景。
- 状态必须同时显示中文文字，不得只靠颜色表达。
- 悬停、选中和禁用状态必须能被清楚区分；选中态的对比度必须明显强于悬停态。
- 禁止装饰渐变、光斑和无业务含义的高饱和颜色。

### 2.2 字体

- 字体栈统一使用 `Inter`、`SF Pro Text`、`PingFang SC`、`Microsoft YaHei` 和系统无衬线回退。
- `12px`：表格、详情、筛选、状态和说明；`13px`：菜单、按钮和重要标签；`14px`：模块标题；`16px`：产品品牌上限。
- 普通业务页面禁止使用 `18px+` 大标题；字间距固定为 `0`，数字表格使用等宽数字特性。
- 清晰度优先通过文字颜色、背景对比、间距和层级建立，不依赖粗大字重。正文和业务数据默认使用常规字重，关键标题和选中项最多使用克制的半强调字重，保持轻盈、精细。

### 2.3 间距与尺寸

- 基础间距序列：`4 / 6 / 8 / 10 / 12 / 14 / 16 / 20 / 24px`。
- 标准尺寸：顶部全局栏 `52px`、工作台页签栏 `38px`、导航项 `36px`、按钮和状态分类 `28px`、紧凑表格行 `36px`、详情标题栏 `44px`。
- 详情栏桌面宽 `328px`、窄桌面宽 `292px`、折叠轨道 `40px`。
- 普通控件圆角 `6px`，不得超过 `8px`；除圆形状态点外不使用胶囊化外观。

## 3. 全局工作台与菜单

全局工作台固定由左侧品牌与层级导航、顶部模块信息、单行工作台页签、中间业务内容、右侧上下文详情组成；必要的事件或日志位于业务内容底部。禁止增加第二套全局导航或重复的大页面标题。

菜单必须支持层级、展开收缩、子级标识、当前选中和父级高亮，并遵守：

- 一级、二级、三级使用稳定递增的缩进，视觉位置不得重合；含三级菜单的二级标题仍保持完整二级缩进。
- 当前项使用清晰的强调文字、浅强调背景和必要的边缘标记；悬停使用中性背景，且不得比选中态更醒目。
- 选中子项时，父级保持适度高亮；切换其他一级菜单时，未选中的展开组应收缩。
- 平台名称可返回运营中控台；品牌文字、左右收展等结构性控件悬停时不使用玻璃浮层或悬浮卡片效果。
- 图标控件使用可访问名称；熟悉图标无需重复显示原生浏览器文字气泡。

## 4. 标准页面骨架

运营中控台可采用地图加详情的特殊布局。其他业务对象、单据、策略和执行记录页面默认使用以下紧凑列表结构：

1. 状态分类 `.status-segment-bar`；
2. 搜索过滤 `.list-filter-bar`；
3. 全局操作 `.list-action-bar`，无全局操作时省略；
4. 表格 `.record-table-section`；
5. 事件区 `.event-log-section`，按业务需要；
6. 底部信息 `.module-footer`；
7. 右侧详情 `.detail-rail`。

新增列表模块必须复用 `RecordTable`、统一页面状态结构和以上类名，不复制相似页面形成分叉。

经营分析、指标分析和管理者决策类页面属于分析型页面，不属于业务对象列表页。分析型页面的主目标是帮助用户理解经营结果、发现异常、判断原因和下钻验证，默认不显示状态分类、关键词查询、状态筛选和表格主导布局。分析型页面应优先展示核心指标、链路解释、质量提示和必要下钻入口；指标观测、计算记录和明细表只能作为相对弱化的折叠、详情或专门记录页面出现。

经营指标类分析页面只能表达统一经营数据池的刷新和展示，不得让每个分析页呈现为独立计算入口。统计周期选择与“更新经营数据”应刷新同一批指标结果，经营指标总览、财务表现、服务经营和过程诊断只按角色视角消费同一数据池。

地图页面使用 `.map-stage`、`.zone-canvas-new`、`.map-floating-actions` 和 `.legend-new`；工具不得遮挡关键对象，图例不得挤压详情栏，地图语义优先于装饰。

## 5. 筛选、状态与操作

- 搜索过滤必须包含筛选字段、`查询` 和 `重置`，并与状态分类联动。
- 点击状态分类时，以该分类重新计算列表并清空临时筛选，避免旧条件锁定数据。
- 菜单首次进入默认展示全部数据；再次返回可恢复该页面上次状态。
- 全局操作放全局功能区；单条记录操作固定在表格最右侧，并复用 `RowActionButton` 和 `RowActionGroup`。
- 行操作只显示当前状态可执行动作；无状态动作但可查看时统一显示 `查看详情`。
- 状态值复用 `StatusValue`，英文枚举必须通过 `valueDictionary` 显示中文。

## 6. 表格、页签与详情

- 表头、行高、内边距和分隔线全站统一；当前行使用 `--accent-soft`，悬停行使用 `--surface-muted`。
- 关键识别字段靠前、数字对齐稳定、长文本省略，完整信息由详情承接。
- 工作台页签单行展示，溢出横向滚动，不换行；当前页签用文字主色和底部细线表达，不做卡片或胶囊。
- 右侧详情跟随当前菜单和选中记录，支持折叠和展开；`查看详情` 应选中记录并在必要时展开详情。
- 详情标题统一 `.panel-title`；简单对象用单列 `Descriptions`，字段较多时用 `Tabs + Descriptions`。
- 表格列名、详情字段名、结构化详情 key、状态和枚举值必须通过统一字段展示服务进入字段字典，不允许页面局部自行维护第二套字段中文或依赖英文兜底。
- 详情为空时显示统一空状态；内容较长时内部滚动，不推动全局布局。
- 中间内容、表格和详情的上下边界必须保持对齐。
- 详情中的复杂字段（数组、对象、快照、配置、计算过程、字段解释、来源说明）必须整行块级展示，不得塞入 `Descriptions` 的右侧窄内容格。
- 对象、快照、配置、事件负载和复杂数组不得直接格式化为连续长文本；统一使用“摘要先行、结构化键值、嵌套分组、默认收起、按需展开”的复杂详情组件。
- 复杂详情的层级通过缩进、细分隔线和间距表达，不使用卡片套卡片、字体放大或厚重背景；位置、路径步骤、状态时间线等已有专用表达优先于通用复杂详情。

## 7. 工作台状态

页面状态应结构化维护，可保留状态分类、搜索条件、分页、当前记录和详情显示状态。工作台切换不得明显占用主业务区，后续新增业务页面默认接入统一状态结构，不把状态散落在临时变量中。

## 8. 交互状态完整性

新增或修改控件必须覆盖适用的默认、悬停、聚焦、选中、禁用、加载、空数据、错误、折叠和展开状态。键盘聚焦必须可见，并落在具体可操作项上，不给整个导航容器绘制粗重焦点框。

## 9. 新模块与模型接入协议

任何开发者或模型新增前端模块时必须：

1. 按 `AGENTS.md` 判断并读取必要规则；
2. 查看现有页面、相近流程和 `src/styles.css`，确认属于地图页、标准列表页或现有特殊工作台；
3. 复用现有页面模板、组件、类名和令牌，不仅凭提示词重新想象风格；
4. 接入统一状态分类、查询重置、分页、选中记录、详情和字段字典；
5. 新增可展示字段、嵌套字段或枚举值时同步字段展示合同检查，确保不会在表格、详情或配置快照中显示内部英文；
6. 仅在现有标准无法表达通用需求时扩展本文件及 `src/styles.css` 令牌；
7. 在 `1280 x 720` 与 `1440 x 900` 真实验证，并检查中文、布局、交互和控制台错误；
8. 无法完成真实页面验证时不得声称完成；发现规则冲突时暂停说明，不创建第三套规则。

任务单和业务执行记录页面必须默认带有最近事件区，除非业务文档明确该对象没有事件语义。新增任务单不得只接入状态分类、筛选和表格而省略事件区；事件区用于解释任务从何而来、是否由策略生成、是否无候选、是否创建成功以及后续关键动作。

有人工触发价值的业务对象或任务单，页面必须提供对应的全局人工触发入口。人工触发和自动触发只能调用同一套业务服务，不得由前端按钮拼装一套与自动化不同的创建逻辑。

策略配置、策略执行和策略结果页面必须展示策略关键参数、执行范围、候选对象、生成对象和配置快照。配置快照不得以长 JSON 文本堆叠，应通过字段字典和复杂详情组件做结构化摘要。

## 10. 完成验收

- [ ] 页面骨架是否来自标准模板并复用现有组件和令牌？
- [ ] 状态分类、查询重置、操作、表格、详情和底部信息是否形成闭环？
- [ ] 业务字段、按钮、筛选、状态和枚举是否全部中文显示？
- [ ] 新增字段是否通过统一字段展示服务和字段展示合同检查？
- [ ] 字号、字重、间距、缩进、圆角、边界和状态色是否清晰一致？
- [ ] 第一视觉感受是否精致、轻盈，且没有用粗大字重或厚重色块换取清晰度？
- [ ] 默认、悬停、聚焦、选中、禁用、空状态和收展状态是否完整？
- [ ] `1280 x 720` 与 `1440 x 900` 是否无重叠、遮挡和不可操作区域？
- [ ] 地图、表格、事件区和详情边界是否对齐，控制台是否无新增错误？
- [ ] 是否避免一次性样式、组件分叉和业务关系硬编码？

## 11. 禁止事项

- 单页硬编码新主色、字体、圆角或阴影；
- 复制已有组件后仅改样式形成分叉；
- 卡片套卡片、页面区块悬浮卡片化；
- 大标题、营销 Hero、装饰渐变、光斑和无业务含义插画；
- 为视觉整齐隐藏必要业务字段或操作；
- 用颜色代替文字状态，或直接展示英文枚举和内部字段名；
- 修改页面后不检查详情、空状态、事件区和窄桌面；
- 为当前页面便利写死对象关系或状态逻辑。

# v036.3 服务订单自动限时分配能力

## 版本定位

本次作为 `v36.x` 的小版本迭代，修正服务订单 `待分配 Robotaxi` 阶段的底层业务模型。

核心目标：服务订单本身具备“人工分配 Robotaxi”和“自动限时分配 Robotaxi”两种入口。模拟运行只通过统一世界时间驱动服务订单已有能力，不在模拟运行中另写匹配、重试或取消逻辑。

## 当前问题

1. 现有实现把 `订单匹配重试` 和 `订单分配等待超时` 拆成两个时间作业，用户难以理解。
2. 匹配失败后的自动化更像“等待一段时间后再匹配一次”，不符合真实叫车业务中“尽可能快速分配”的目标。
3. `CANCELLED` 在时间作业里表示作业撤销，在服务订单里表示订单取消，前端语义容易混淆。
4. `order_matching_retry_seconds` 与 `order_assignment_wait_seconds` 在运行时配置映射中指向同一条工作流规则，配置语义不够独立。

## 目标设计

### 服务订单能力

- 人工分配 Robotaxi：保留 `ORDER_MATCHING_EXECUTE`，人工或单次动作调用时只执行一次匹配。
- 自动限时分配 Robotaxi：订单确认叫车后创建一个 `订单自动分配` 时间作业；该作业在最大等待时长内按重试间隔推进匹配，成功则完成，超时则取消订单。

### 时间作业语义

- 新增作业类型：`ORDER_AUTO_ASSIGNMENT`。
- 新增动作类型：`ORDER_AUTO_ASSIGNMENT_TICK`。
- `ORDER_AUTO_ASSIGNMENT` 是服务订单自动分配会话，不是取消订单作业，也不是单次重试作业。
- 作业 `COMPLETED` 表示自动分配会话已产生终态：匹配成功或超时取消。
- 作业 `CANCELLED` 展示为“已撤销”，用于业务单据已进入其他生命周期、该作业无需继续推进。

### 配置语义

- `assignment_retry_interval_seconds`：自动分配重试间隔，默认 1 秒。
- `assignment_max_wait_seconds`：自动分配最大等待时长，默认 60 秒。
- 保留旧字段兼容：`order_matching_retry_seconds`、`assignment_wait_timeout_seconds`、`order_assignment_wait_seconds`。

## 验收标准

- 客户确认叫车后，服务订单进入自动分配中，并生成一个 `订单自动分配` 时间作业。
- 自动分配作业按统一世界时间推进，每次到达重试间隔时调用同一底层匹配服务。
- 匹配成功时订单进入履约，自动分配作业完成。
- 超过最大等待时长仍未匹配成功时，订单取消，自动分配作业完成。
- 人工分配入口仍可调用 `ORDER_MATCHING_EXECUTE` 执行一次匹配，不被自动分配逻辑覆盖。
- 时间作业 `CANCELLED` 在前端显示为“已撤销”。
- 字段字典、代码字典、验证脚本、bundle、提交和 tag 同步完成。

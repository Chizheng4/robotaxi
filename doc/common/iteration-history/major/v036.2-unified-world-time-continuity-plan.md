# v036.2 统一世界时间连续性修复

## 版本定位

本次作为 `v36.x` 的小版本收口，修复模拟运行时间轴的底层连续性问题。

目标不是新增业务功能，而是把已经确认的统一时间世界规则落到代码：

- 统一世界时间是业务单据、状态时间线、成本、收入和指标统计的事实时间。
- Tick 只是调度间隔，不是业务事实边界。
- 计划运行窗口结束只停止新增需求和供给触发，收尾阶段继续推进已有业务动作。
- 下一次模拟运行应从上一运行的计划窗口结束时间继续触发，不应被上一运行的收尾完成时间推迟。
- 重置模拟数据后，第一次新建模拟运行必须回到 `Day 1 00:00:00`。

## 当前问题

1. `initSimulationRun` 继承上一运行时间时优先使用 `simulation_end_seconds`，该字段包含收尾阶段，可能导致下一次运行错过计划窗口结束到收尾完成之间的触发点。
2. 前端创建模拟运行时基于渲染闭包中的 `simulationRuns` 计算上一运行；重置后若状态尚未完成刷新，可能继承旧运行。
3. 重置模拟数据时没有显式重置模拟运行编号和事件编号计数器，容易造成用户误判“第一次运行不是全新时间线”。
4. 架构规则尚未明确“连续运行起点取上一计划窗口结束，实际结束只记录收尾事实”的硬约束。

## 完成记录

- `simulationEngine.initSimulationRun` 改为优先继承上一运行 `planned_simulation_end_seconds`，实际结束秒只作为旧数据兜底。
- `current_global_tick` 改为由统一世界秒和 `tick_seconds` 推导。
- `simulationActions.createSimulationRun` 改为使用最新 React state 中的运行列表选择上一运行，避免重置后旧闭包污染新运行。
- `resetRuntime` 同步调用 `resetSimulationCounters`，重置后模拟运行和事件编号从全新状态开始。
- `07-simulation-runtime-architecture-rules.md` 增加计划窗口、实际结束和连续运行起点规则。
- `verify-simulation-continuity.mjs` 增加计划窗口连续性验证。

## 验收标准

- 首个模拟运行无上一运行时从 `Day 1 00:00:00` / `0` 秒开始。
- 上一运行收尾超过计划结束时，下一运行从 `planned_simulation_end_seconds` 开始，而不是 `simulation_end_seconds`。
- 下一运行的 `current_global_tick` 从统一世界秒和 `tick_seconds` 推导，避免继承旧 Tick 偏差。
- 重置模拟数据后模拟运行计数器、事件计数器和时间轴回到全新状态。
- `scripts/verify-simulation-continuity.mjs` 覆盖以上合同。
- 修改 `src/main.jsx` 后重新编译 bundle，语法检查和提交前检查通过。

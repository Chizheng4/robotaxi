# v028.1 模拟工作流时效与业务状态时间线方案

## 1. 文档状态

- 版本：`v028.1`
- 阶段：实现完成
- 状态：已完成并通过验收
- 编码：已完成
- 历史方案：本文件
- 后续衔接方案：`operating-metrics-model-pending-plan.md`

本方案解决自动运营模拟业务单据的模拟创建时间、状态变更时间、跨单据因果时间和行驶时间计算问题，为后续经营指标模型提供可信的业务时间事实。

## 2. 已确认业务原则

1. 自动运营模拟继续使用当前 Tick、WorkflowEngine 和 ExecutionEngine 快速执行。
2. 时效配置不控制执行引擎等待，不降低模拟速度。
3. SimulationRun 完成并排空全部工作流后，自动执行独立的业务时效计算。
4. 时效配置与 SimulationPolicy 完全分离，拥有独立对象、页面和版本。
5. 人工可在已完成 SimulationRun 上重新计算，并看到进度、成功或错误结果。
6. 业务状态、工作流动作和状态边必须来自统一工作流定义，不允许前端自由输入。
7. 运营行驶记录与履约行驶记录的移动耗时按 Route 实际移动步数乘以单 Cell 行驶时间计算。
8. 跨单据同步状态由同一个因果事件计算一次，再投影到相关单据，禁止重复累加。
9. 业务单据展示最新成功计算结果，历史计算运行和配置快照保留。
10. 当前经营指标方案调整为后续待执行方案，v028.1 先建立指标所需的业务时间基础。

## 3. 目标与非目标

### 3.1 目标

1. 建立独立、版本化、可配置的工作流动作时长模型。
2. 在快速模拟完成后重建可解释、精确到秒的业务状态时间线。
3. 支持源头单据、派生单据和跨单据同步状态的因果计算。
4. 将计算结果保存到业务单据，并保留完整状态变更历史。
5. 在 SimulationRun 和业务单据详情中展示计算状态、时间线和错误。
6. 保证相同业务数据、Route、配置版本和计算算法得到相同结果。

### 3.2 非目标

- 不让 ExecutionEngine 按配置时长真实等待。
- 不改变 SimulationRun 的 Tick 推进速度。
- 不在本版本实现经营指标计算。
- 不使用真实审计时间替代模拟业务时间。
- 不在确认前更新字段字典或编码。
- 不用前端临时逻辑独立推算时间。

## 4. 核心概念与统一命名

### 4.1 模拟工作流时效配置

- 对象英文名：`SimulationWorkflowTimingProfile`
- 对象中文名：模拟工作流时效配置
- 页面名称：工作流时效配置
- 作用：为正式工作流状态边配置动作耗时。

### 4.2 工作流时效规则

- 对象英文名：`WorkflowTimingRule`
- 对象中文名：工作流时效规则
- 作用：定义一个业务对象从当前状态，经指定动作进入下一状态所需的时间。

唯一业务键：

```text
business_object_type
+ from_status
+ action_type
+ to_status
+ result_type（适用异常分支时）
```

### 4.3 业务时效计算运行

- 对象英文名：`BusinessTimingCalculationRun`
- 对象中文名：业务时效计算运行
- 操作名称：计算业务时效 / 重新计算业务时效
- 作用：针对一个已完成 SimulationRun，使用一份时效配置快照生成业务时间线。

### 4.4 模拟状态变更记录

- 字段英文名：`simulation_status_transition_history`
- 字段中文名：模拟状态变更记录
- 作用：按顺序保存业务对象每次状态进入时间、来源动作和配置时长。

### 4.5 原始模拟时间与计算模拟时间

为了不覆盖快速模拟事实，两套时间必须明确分开：

|时间类型|字段风格|含义|用途|
|---|---|---|---|
|原始模拟时间|`simulation_*`|当前快速模拟在 Tick 中记录的发生时间|模拟执行追踪、计算锚点|
|计算模拟时间|`calculated_simulation_*`|SimulationRun 完成后根据工作流和时效配置计算的业务时间|经营分析、状态时间线、时效指标|
|真实审计时间|`created_at`、`completed_at` 等|程序真实写入和执行时间|审计、性能排查|

后续经营指标必须优先使用 `calculated_simulation_*`；未成功计算的单据不得悄悄回退到真实时间。

## 5. 总体架构

```text
快速自动模拟
SimulationLoop → WorkflowEngine → ExecutionEngine
         ↓ 保存业务对象、Route、SimulationEvent、原始模拟时间
SimulationRun RUNNING → DRAINING → COMPLETED
         ↓ 自动触发
BusinessTimingCalculationRun
         ↓ 读取
WorkflowTimingProfile 快照 + 业务对象 + Route + 工作流因果关系
         ↓ 计算
业务状态时间线 + 计算模拟里程碑 + 校验结果
         ↓ 投影
SimulationRun 计算状态 / 业务单据状态时间线 Tab / 后续经营指标
```

计算器是模拟完成后的独立模块，不嵌入每个 Tick，也不改变现有执行引擎的运行方式。

## 6. 自动与人工计算流程

### 6.1 自动计算

触发条件：

1. SimulationRun 状态进入 `COMPLETED`；
2. DRAINING 已结束，不存在待执行工作流；
3. 存在一份 `ACTIVE` 的工作流时效配置；
4. 本 SimulationRun 尚无成功的自动计算运行。

执行过程：

```text
SimulationRun COMPLETED
→ 创建 BusinessTimingCalculationRun（QUEUED）
→ 保存配置快照
→ 状态 CALCULATING
→ 构建跨单据因果图
→ 按依赖顺序计算
→ 校验并写入单据
→ SUCCEEDED / PARTIALLY_SUCCEEDED / FAILED
```

### 6.2 人工重新计算

人工入口仅对 `COMPLETED` 的 SimulationRun 默认开放。

点击后弹出确认框，展示：

- SimulationRun 编号和时间轴；
- 当前生效配置及版本；
- 上一次成功计算运行；
- 本次将创建新计算版本；
- 不修改原始模拟事件和真实审计时间；
- 最新成功结果将成为业务单据当前展示结果。

`STOPPED` 或 `FAILED` 的 SimulationRun 默认不自动计算。人工计算时必须显示“业务数据可能不完整”，结果最多标记为 `PARTIALLY_SUCCEEDED`。

### 6.3 长任务体验

计算必须异步执行，不阻塞页面。用户离开页面后继续计算。

前端持续展示：

```text
等待计算 → 计算中 37% → 计算成功
```

进度至少包含：

- 已处理业务对象数 / 总对象数；
- 已处理状态边数 / 总状态边数；
- 当前对象类型；
- 开始真实时间；
- 已运行时长；
- 错误数量。

## 7. 配置模型设计

### 7.1 SimulationWorkflowTimingProfile 字段提案

以下字段状态均为 `PROPOSED`，确认后必须同步两个字段字典。

|英文字段|中文名称|性质|说明|
|---|---|---|---|
|workflow_timing_profile_id|工作流时效配置编号|持久化|唯一编号|
|profile_name|配置名称|持久化|用户可识别名称|
|profile_version|配置版本|持久化|不可变版本号|
|profile_status|配置状态|持久化|DRAFT、ACTIVE、ARCHIVED|
|description|配置说明|持久化|适用场景|
|timing_rules|时效规则列表|持久化|WorkflowTimingRule 集合|
|created_at|创建时间|审计|真实创建时间|
|updated_at|更新时间|审计|真实更新时间|
|activated_at|生效时间|审计|真实生效时间|

同一时间只能存在一份 ACTIVE 配置。SimulationRun 完成时使用当时 ACTIVE 配置创建不可变快照。

### 7.2 WorkflowTimingRule 字段提案

|英文字段|中文名称|性质|说明|
|---|---|---|---|
|workflow_timing_rule_id|工作流时效规则编号|持久化|唯一编号|
|business_object_type|业务对象类型|持久化|来源于工作流注册表|
|from_status|变更前状态|持久化|来源于正式状态枚举|
|action_type|工作流动作类型|持久化|来源于 WorkflowEngine|
|to_status|变更后状态|持久化|动作成功后的正式状态|
|result_type|动作结果类型|持久化|成功、失败或特定异常分支|
|duration_mode|耗时模式|持久化|FIXED_DURATION、PER_CELL_DURATION|
|fixed_duration_seconds|固定操作时长（秒）|持久化|固定耗时模式使用|
|seconds_per_cell|单 Cell 行驶时长（秒）|持久化|行驶模式使用|
|minimum_duration_seconds|最小操作时长（秒）|持久化|计算保护下限|
|rule_status|规则状态|持久化|ENABLED、DISABLED|
|display_order|显示顺序|持久化|前端稳定排序|

### 7.3 耗时模式

第一阶段只支持：

|英文值|中文名称|公式|
|---|---|---|
|FIXED_DURATION|固定操作时长|`action_end = action_start + fixed_duration_seconds`|
|PER_CELL_DURATION|按 Cell 计算行驶时长|`action_end = action_start + movement_step_count × seconds_per_cell`|

暂不加入随机分布。相同数据和配置必须得到完全相同结果。

## 8. 配置前端设计

### 8.1 菜单位置

```text
自动运营模拟
├── 模拟规则配置
├── 工作流时效配置
└── 模拟运行管理
```

工作流时效配置与 SimulationPolicy 分离，不放入模拟规则配置详情。

### 8.2 页面结构

页面使用现有标准列表骨架：

1. 配置状态分类；
2. 配置搜索和筛选；
3. 新建、复制、启用、归档操作；
4. 时效配置列表；
5. 右侧配置详情。

详情按业务对象分组展示工作流状态边：

|业务对象|当前状态|工作流动作|下一状态|耗时模式|时长配置|状态|
|---|---|---|---|---|---|---|
|运营准入任务|待分配|分配 Worker|待检查|固定时长|3 秒|启用|

状态、动作和下一状态均只读，由统一工作流注册表提供；用户只配置耗时模式和时长。

### 8.3 配置校验

- 每条可执行工作流状态边必须有且只有一条启用规则；
- 固定时长不得小于 0；
- 行驶动作必须使用 PER_CELL_DURATION；
- 非行驶动作不得配置 seconds_per_cell；
- 配置存在重复边时禁止启用；
- 工作流发生变化后，旧配置必须显示“待补齐状态边”。

## 9. 工作流注册表要求

当前 WorkflowEngine 主要保存 `fromState + actionType`，但时效配置需要完整的：

```text
objectType + fromState + actionType + toState + resultType
```

因此进入编码前必须先形成唯一工作流状态边注册表。配置页面和计算器共同读取该注册表，禁止各自维护状态映射。

### 9.1 当前已识别冲突

|对象|当前问题|v028.1 目标|
|---|---|---|
|DeploymentTask|创建后当前代码为 WAITING_ROUTE|按已确认方案改为 WAITING_START|
|RouteExecution|创建后为 WAITING_ROUTE|保持 WAITING_ROUTE|
|DeploymentTask / RouteExecution|ROUTE_PLAN 同时推动两个对象|注册为一个因果动作、两个状态投影|
|ServiceOrder|状态枚举多于自动工作流实际使用状态|只配置实际可达边，未接入状态不得伪造|
|Trip|一个 TRIP_STEP_EXECUTE 覆盖创建、状态推进和移动|拆分为可识别的逻辑状态边，不一定拆分执行 handler|
|SimulationPolicy.execution_time_config|已有少量 Tick 耗时配置|迁移为独立时效配置，避免两套来源|

在工作流状态边未统一前，不允许开发配置页面。

## 10. 业务对象范围

### 10.1 核心状态单据

以下对象保存完整 `simulation_status_transition_history`：

1. ReadinessTask / 运营准入任务；
2. DeploymentTask / 运营投放任务；
3. RouteExecution / 运营行驶记录；
4. ServiceOrder / 服务订单；
5. Trip / 履约行驶记录。

### 10.2 派生执行记录

以下对象记录计算模拟创建、开始和完成时间，但不强制具有完整状态机：

- DemandSimulationRun / 需求模拟执行；
- RoutePlanningRun / 路径规划执行；
- PricingStrategyRun / 定价执行；
- PricingDecision / 定价结果；
- OrderMatchingRun / 订单匹配执行；
- OrderMatchingDecision / 订单匹配结果；
- Route / 路径。

派生执行记录的时间必须来自创建它们的工作流动作，不能从自身真实 `created_at` 推导。

## 11. 运营准入任务时间线

### 11.1 正常流程

|顺序|当前状态|动作|下一状态|时间计算|
|---|---|---|---|---|
|0|无|READINESS_TASK_CREATE|WAITING_ASSIGNMENT|源头触发时间 T0|
|1|WAITING_ASSIGNMENT|READINESS_TASK_ASSIGN|WAITING_CHECK|T0 + 分配 Worker 时长|
|2|WAITING_CHECK|READINESS_TASK_START|CHECKING|上一时间 + 开始检查时长|
|3|CHECKING|READINESS_TASK_PASS|COMPLETED|上一时间 + 准入检查时长|

示例：

```text
T0 = Day 1 08:00:00
分配 Worker = 3s
开始检查 = 2s
准入检查 = 120s

待分配  Day 1 08:00:00
待检查  Day 1 08:00:03
检查中  Day 1 08:00:05
已完成  Day 1 08:02:05
```

失败、取消和检查不通过分支必须使用独立状态边和结果类型，不复用正常完成时长。

## 12. 运营投放与运营行驶时间线

### 12.1 创建依赖

按已确认业务规则：

```text
运营准入完成
→ 触发创建运营投放任务
→ DeploymentTask 进入 WAITING_START
→ 同一创建动作生成 RouteExecution
→ RouteExecution 进入 WAITING_ROUTE
```

直接时间关系：

```text
DeploymentTask.WAITING_START.entered_at
= RouteExecution.simulation_created_at
= RouteExecution.WAITING_ROUTE.entered_at
```

运营准入完成时间是间接上游；创建投放任务动作完成时间是直接 T0。

### 12.2 路径规划同步变更

|来源状态|动作|结果状态|时间|
|---|---|---|---|
|RouteExecution.WAITING_ROUTE|ROUTE_PLAN|RouteExecution.MOVING|T0 + 路径规划时长|
|DeploymentTask.WAITING_START|同一 ROUTE_PLAN 因果事件|DeploymentTask.MOVING|与 RouteExecution.MOVING 完全相同|

ROUTE_PLAN 只计算一次，生成一个状态变更事件，再投影到两个对象。

### 12.3 行驶与到达

```text
movement_step_count = max(route.route_steps.length - 1, 0)
travel_duration_seconds = movement_step_count × seconds_per_cell
arrived_at = moving_at + travel_duration_seconds
```

Route 起点 Cell 不算一次移动。只有 Cell 之间的迁移次数计入移动步数。

|来源状态|动作|同步结果|
|---|---|---|
|RouteExecution.MOVING|ROUTE_EXECUTION_STEP|RouteExecution.ARRIVED|
|DeploymentTask.MOVING|同一行驶完成事件|DeploymentTask.ARRIVED|
|ARRIVED|ARRIVAL_CONFIRM|两个对象进入 COMPLETED|

异常到达、暂停、失败、取消及重规划必须保留独立状态变更记录。

## 13. 服务订单与履约行驶时间线

### 13.1 订单主链

```text
需求触发
→ 创建 ServiceOrder
→ 定价
→ 客户确认叫车
→ 订单匹配
→ 创建 Trip
→ 接驾路径规划与行驶
→ 乘客上车
→ 送达路径规划与行驶
→ 结算
→ 支付
→ 完成
```

每个派生记录创建时间等于创建它的动作完成时间：

|派生对象|来源动作|计算模拟创建时间|
|---|---|---|
|PricingStrategyRun / PricingDecision|PRICING_EXECUTE|定价动作完成时间|
|OrderMatchingRun / OrderMatchingDecision|ORDER_MATCHING_EXECUTE|匹配动作完成时间|
|Trip|TRIP_STEP_EXECUTE 的创建分支|Trip 创建动作完成时间|
|RoutePlanningRun / Route|ROUTE_PLAN|路径规划动作完成时间|

### 13.2 跨单据同步原则

- 订单匹配成功事件同时更新 ServiceOrder 和 Robotaxi；
- Trip 创建事件同时写入 ServiceOrder.trip_id；
- Trip 到达接驾点时同步订单进入对应等待上车状态；
- 乘客上车事件同步 Trip 与 ServiceOrder；
- Trip 到达目的地事件同步 ServiceOrder 进入结算阶段；
- 支付完成事件写入 ServiceOrder 的计算模拟支付完成时间。

每个同步事件只有一个计算时间，所有关联对象引用该时间。

## 14. 路径与行驶计算规则

### 14.1 两类行驶对象

- RouteExecution：运营任务行驶；
- Trip：服务订单接驾和送达行驶。

两者使用同一个 `PER_CELL_DURATION` 计算器，但可配置不同的 `seconds_per_cell` 规则。

### 14.2 多段路径

Trip 的接驾 Route 和送达 Route 分开计算：

```text
pickup_travel_seconds
= pickup_route_movement_steps × pickup_seconds_per_cell

destination_travel_seconds
= destination_route_movement_steps × destination_seconds_per_cell
```

### 14.3 重规划

重规划时按 `route_history` 分段计算：

1. 原 Route 只计算已实际经过的移动步数；
2. 重规划动作增加配置时长；
3. 新 Route 从当前 Cell 开始计算剩余移动步数；
4. 不允许用最终 Route 覆盖之前已经发生的行驶时间。

### 14.4 缺失 Route

需要行驶但找不到有效 Route 或 route_steps 时：

- 当前对象计算标记失败；
- 记录 `ROUTE_DATA_MISSING`；
- 不用 estimated_distance 或默认 1 Cell 伪造；
- 其他无依赖对象可继续计算，整个运行标记为 PARTIALLY_SUCCEEDED。

## 15. 跨单据因果图

计算器不能逐张单据孤立计算，必须先构建有向无环因果图。

### 15.1 节点

一个节点代表一个业务动作或状态进入事件。

### 15.2 边

边代表时间依赖：

```text
上游动作完成
→ 下游对象创建
→ 下游状态进入
```

### 15.3 时间规则

```text
action_started_at = max(
  current_status_entered_at,
  all_prerequisite_completed_at
)

action_completed_at
= action_started_at + configured_duration_seconds
```

多个前置条件并行完成时，以下游所有前置条件中的最晚完成时间为开始时间，不能简单把并行耗时全部相加。

### 15.4 因果来源字段提案

|英文字段|中文名称|说明|
|---|---|---|
|source_business_object_type|来源业务对象类型|直接上游对象类型|
|source_business_object_id|来源业务对象编号|直接上游对象编号|
|source_transition_id|来源状态变更编号|直接触发本事件的状态变更|
|causation_id|因果链编号|串联同一业务因果链|
|correlation_id|业务关联编号|串联同一订单、任务或履约上下文|

## 16. 计算算法

### 16.1 输入

- SimulationRun 及时间轴；
- 快速模拟生成的全部业务对象；
- SimulationEvent 和业务事件；
- Route、route_steps、route_history；
- 工作流状态边注册表；
- SimulationWorkflowTimingProfile 快照。

### 16.2 步骤

1. 校验 SimulationRun 是否允许计算；
2. 冻结业务数据版本和时效配置快照；
3. 识别源头触发对象及原始模拟锚点；
4. 构建业务对象关联关系；
5. 将实际执行动作映射到工作流状态边；
6. 构建跨单据因果图；
7. 检测循环依赖、孤立节点和重复状态边；
8. 按拓扑顺序计算动作开始和完成时间；
9. 对行驶动作按 Route 移动步数计算；
10. 生成每个对象的模拟状态变更记录；
11. 投影常用计算模拟里程碑字段；
12. 执行跨对象时间一致性校验；
13. 保存计算结果和错误；
14. 将最新成功运行设为当前有效结果。

### 16.3 确定性

计算不读取当前真实时间作为模拟结果，不使用无种子的随机数。相同输入、配置快照和算法版本必须得到完全相同的状态时间线。

## 21. 重新计算与版本规则

1. 每次计算创建新的 BusinessTimingCalculationRun；
2. 历史计算运行不删除、不覆盖；
3. 业务单据只指向一个当前有效计算运行；
4. 新计算成功后才切换当前有效结果；
5. 新计算失败时继续保留上一次成功结果；
6. 指标结果必须记录使用的业务时效计算运行编号；
7. 时效配置归档后，其历史快照仍可复现旧结果。

## 22. 错误与校验设计

### 22.1 错误类型提案

|英文值|中文名称|处理方式|
|---|---|---|
|TIMING_RULE_MISSING|缺少时效规则|当前对象失败|
|WORKFLOW_EDGE_INVALID|工作流状态边无效|当前对象失败|
|DEPENDENCY_MISSING|缺少上游依赖|当前对象失败|
|DEPENDENCY_CYCLE|存在循环依赖|整次计算失败|
|ROUTE_DATA_MISSING|缺少路径数据|相关行驶对象失败|
|ROUTE_STEP_INVALID|路径移动步数无效|相关行驶对象失败|
|SIMULATION_ANCHOR_MISSING|缺少模拟时间锚点|当前因果链失败|
|DUPLICATE_TRANSITION|重复状态变更|当前对象失败|
|STATUS_SEQUENCE_INVALID|状态顺序无效|当前对象失败|
|TIMING_RESULT_CONFLICT|跨单据时间不一致|关联对象失败|

### 22.2 一致性校验

- 每个对象状态时间必须单调不减；
- 下一状态时间等于动作开始时间加配置时长；
- 子单据创建时间不得早于来源动作完成时间；
- DeploymentTask.MOVING 与 RouteExecution.MOVING 时间必须相同；
- ServiceOrder 与 Trip 的共享里程碑时间必须相同；
- 完成时间不得早于创建时间；
- 行驶时长必须与移动步数和单 Cell 时长一致；
- 当前有效计算结果必须来自成功或部分成功运行；
- 配置快照必须覆盖所有实际发生的工作流状态边。

## 23. 性能与存储原则

1. 计算只在模拟完成后执行，不进入 Tick 热路径；
2. 计算按对象和因果链分批处理，避免阻塞浏览器主线程；
3. 进度与中间状态定期持久化，页面刷新后可恢复；
4. 完整历史计算运行与状态时间线使用可扩展存储，不回退到 localStorage；
5. 大量时间线在前端采用分页或受控渲染；
6. 自动计算失败不得影响原 SimulationRun 的完成状态；
7. 计算输入必须冻结，防止运行过程中业务对象被修改导致结果漂移。

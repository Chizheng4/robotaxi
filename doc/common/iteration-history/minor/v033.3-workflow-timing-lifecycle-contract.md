# v033.3 工作流时效与业务生命周期时间线小迭代

## 迭代类型

小迭代。

## 目标

把工作流时效配置定义为业务单据生命周期自动化推进的底层耗时合同，而不是模拟完成后的补算工具。

本轮聚焦最小经营闭环的真实模拟基础：

1. 业务单据自身生命周期是底层事实来源；
2. 模拟运行、真实时间自动化、成本、收入和后续指标都只能消费业务单据沉淀的数据；
3. 工作流时效配置必须参与状态变更时间线写入；
4. 状态时间线不得大量出现配置存在但耗时为 `0s` 的错误结果；
5. 工程结构保持简单，避免在模拟运行中增加第二套状态推进逻辑。

## 当前问题

v033.2 已经把状态时间线沉淀入口移动到业务动作服务，但 `withLifecycleStatus()` 默认使用 `0s`，多数动作没有按 `business_object_type + from_status + action_type` 解析工作流时效规则。

因此前端状态时间线会显示大量 `0s`，并进一步影响成本模型对业务操作耗时的计算。

同时字段字典中仍保留旧描述：工作流时效配置只用于模拟完成后的业务时间线计算。这与当前统一时间世界和业务生命周期底座冲突。

## 方案

1. 在 `workflowRuntimeConfig` 中提供可复用的时效规则解析结果，返回规则和耗时。
2. 在 `businessActionService.withLifecycleStatus()` 中集中解析工作流时效规则。
3. 状态变更时间线写入：
   - `simulation_action_started_at` = 当前状态变更时间 - 配置耗时；
   - `simulation_status_changed_at` = 当前业务动作完成时的世界时间；
   - `configured_duration_seconds` = 工作流时效配置换算后的耗时；
   - `workflow_timing_rule_id` = 命中的时效规则编号。
4. 行驶类状态边使用 `seconds_per_cell * movement_step_count`。
5. 投影状态边继承来源配置，继续由同一套工作流时效配置支撑。
6. 更新规则文档，明确时效配置是业务生命周期运行时配置。
7. 增加专项验证，防止状态时间线入口再次退回默认 `0s`。

## 不做范围

1. 不进入指标体系建设。
2. 不重写手工页面旧函数。
3. 不增加第二套模拟状态机。
4. 不改变成本和收入计算模型，只保证其输入数据更正确。

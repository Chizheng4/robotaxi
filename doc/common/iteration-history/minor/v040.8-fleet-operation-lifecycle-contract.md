# v040.8 Fleet Operations 生命周期闭环修复

## 1. 本轮目标

本轮修复 Robotaxi 运维任务单的底层生命周期合同，避免页面按钮临时串联业务动作。

核心原则：

1. 运维触发必须创建任务单；排队也必须是可追踪的任务单。
2. Robotaxi 一旦存在运维阻塞标记，就不能再接受新的服务订单或运营投放。
3. Robotaxi 正在服务订单或其他任务中时，运维任务进入“等待 Robotaxi 可执行”，不显示目的地分配动作。
4. Robotaxi 可被当前任务接管后，任务进入“待 Robotaxi 执行”或“待分配作业人员”。
5. 目的地分配、待行驶、前往目的地、到达、作业、完成必须由服务层推进。
6. 清洁、维护等作业完成后直接完成或失败，不增加“待结果确认”等多余状态。
7. 运维任务完成后，Robotaxi 必须恢复可运营，才能重新参与服务订单和投放匹配。

## 2. 确认后的业务闭环

### 2.1 触发来源

- Robotaxi 直接触发：模拟真实运营中 Robotaxi 自检、客户使用反馈或运营人员在 Robotaxi 管理中人工触发。
- 运维策略触发：策略扫描 Robotaxi，识别需要清洁、充电、维修、故障处理或退役的车辆，并调用同一套任务创建服务。

### 2.2 Robotaxi 执行权判断

- 正在服务订单或更高优先级任务中：创建任务单，状态为“等待 Robotaxi 可执行”，任务单进入 Robotaxi 待执行队列，不占用 `current_task_id`。
- 空闲且已在具备对应能力的运维区域内：任务单直接进入“待分配作业人员”。
- 空闲但不在对应运维区域内：任务单进入“待 Robotaxi 执行”，由用户触发分配清洁站、充电站、维修中心等目的地。

### 2.3 状态主线

- `WAITING_ROBOTAXI_AVAILABLE`：等待 Robotaxi 可执行。
- `WAITING_DESTINATION_ASSIGNMENT` / `WAITING_CHARGING_DESTINATION_ASSIGNMENT` / `WAITING_MAINTENANCE_DESTINATION_ASSIGNMENT`：面向用户展示为“待 Robotaxi 执行”。
- `WAITING_ROUTE`：待行驶，表示目的地已分配，等待生成运营行驶记录。
- `MOVING_TO_OPS_CENTER` / `MOVING_TO_CHARGER` / `MOVING_TO_MAINTENANCE_CENTER` / `MOVING_TO_RETIREMENT_CENTER`：面向用户展示为“前往目的地”。
- 到达后进入“待分配作业人员”或对应任务处理状态。
- 清洁：待分配作业人员 → 清洁中 → 已完成 / 异常失败。
- 维护：待分配资源 → 维护中 → 已完成 / 异常失败。
- 充电：待分配作业人员 → 接入充电中 → 充电中 → 待分配作业人员 → 断开充电中 → 已完成。

## 3. 执行计划

1. 修复 Fleet Operation 任务服务：
   - 排队也创建任务单；
   - 队列保存 `task_id`；
   - Robotaxi 运维标记立即阻塞新服务订单和投放；
   - 目的地分配由服务层返回任务状态变更；
   - 作业开始和完成由服务层推进。
2. 修复 Robotaxi 可运营判断：
   - 投放任务创建必须调用 Robotaxi 可投放判断；
   - 服务订单匹配必须排除运维阻塞车辆；
   - 服务订单支付释放 Robotaxi 时，如果存在待执行运维任务，不恢复可运营，而是激活运维任务。
3. 修复前端动作：
   - 目的地分配、生成行驶记录、作业开始、作业完成都调用服务层；
   - 最近任务事件记录关键动作；
   - 表格使用固定列宽和固定布局，减少表头与内容错位。
4. 修复展示合同：
   - 状态中文改为“待 Robotaxi 执行”“待行驶”“前往目的地”等用户语言；
   - 运维任务表格派生展示 Robotaxi 当前位置和目的地位置；
   - 同步代码版和文档版字段字典。
5. 补充验证：
   - 新增 v040.8 生命周期合同验证；
   - 更新 v037.2 服务合同到“排队也是任务单”的新语义；
   - 接入 `scripts/check-before-commit.sh`。

# v028.1.5 履约与订单状态边界修复

## 1. 根因

`SETTLING / 结算中` 是 ServiceOrder 的结算状态，但历史 Trip 状态机仍残留同名状态；状态时间线又使用无对象上下文的全局值翻译，导致履约行驶记录可能显示服务订单状态。

## 2. 状态所有权

- Trip 完成：`trip_status = COMPLETED / 已完成`；
- ServiceOrder 结算：`order_status = SETTLING / 结算中`；
- Trip 完成可驱动 ServiceOrder 进入结算中，但不得把 `SETTLING` 写回 Trip。

## 3. 修复

1. 从 TripStatus 和 Trip 状态跳转中移除 `SETTLING`；
2. 本地持久化历史 Trip 的 `SETTLING` 在加载时迁移为 `COMPLETED`；
3. ServiceOrder 实际值同步只读取 Trip 的到达与完成状态；
4. 状态时间线根据对象传入 `task_status`、`execution_status`、`order_status` 或 `trip_status`；
5. 代码与文档字段字典分别维护 `order_status` 和 `trip_status` 的上下文值，禁止依赖全局同名状态猜测业务含义。

## 4. 验证与完成

- TripStatus 不包含 `SETTLING`；
- 历史 Trip `SETTLING` 加载后为 `COMPLETED`；
- Trip 完成仍正确驱动 ServiceOrder 进入 `SETTLING`；
- 履约列表、详情、筛选和状态时间线显示“已完成”；
- 服务订单结算状态继续显示“结算中”；
- 更新双字段字典、bundle 和自动验证；
- 归档方案，提交并创建 `v028.1.5` 标签。

# v032.7 小迭代：统一工作流时效运行时与高速模拟推进

## 版本类型

小迭代。

## 问题识别

1. 模拟世界时间已经改为秒级 Tick，但运行循环仍按真实 500ms 推进 1 个 Tick，24 小时模拟耗时过长。
2. 工作流时效配置已作为前端可配置功能存在，但运行时业务动作仍优先读策略快照旧字段，配置来源不统一。
3. 服务订单匹配失败虽然已有等待重试，但缺少“分配等待上限”，无法表达真实运营中的订单等待超时取消。
4. 供给侧投放只看 Robotaxi 运营时间，未同时约束作业人员工作时间，非工作时间可能产生低价值触发。
5. “自动计算运营节点模拟时间”仍在模拟完成后自动触发，与新的时间驱动运行主路径重复。

## 工程原则

1. 模拟时间推进可以加速，但业务对象仍按 1 秒世界时间逐 Tick 演化。
2. 时间作业只是调度层，到期动作必须继续调用 ExecutionEngine 注册的业务动作。
3. 工作流时效配置是运行时动作耗时主来源，策略快照旧字段只做兼容兜底。
4. 服务订单等待超时是正常业务闭环，超时取消不等同模拟运行失败。
5. 保持最小可运行，不引入新指标体系或未来对象。

## 自动化执行计划

1. 新增工作流运行时配置解析器，按对象、状态和动作从 Active 工作流时效配置解析固定耗时或每 Cell 耗时。
2. 模拟业务 Runtime 注入 Active 工作流时效配置。
3. 业务动作服务改用运行时解析器获取准入检查、Cell 行驶、到达识别、订单分配等待和订单重试耗时。
4. 服务订单 `ROBOTAXI_CALL` 后创建订单分配等待超时时间作业；匹配成功取消未完成的相关等待作业；超时动作取消订单。
5. 模拟控制循环增加 `simulation_speed_config`，用批量 Tick 保持秒级语义并提升真实执行速度。
6. 供给侧投放同时检查作业人员工作时间与 Robotaxi 运营时间。
7. 禁用完成后自动业务时效计算，保留手动重算。
8. 同步字段字典、版本记录、验证脚本、bundle 和提交标签。

## 验收标准

- 1 秒 Tick 语义保留，模拟推进速度可通过策略配置提高。
- 工作流时效配置能影响运行时准入检查、行驶、到达识别和订单等待时间。
- 无车订单在等待超时后进入 `CANCELLED`，不会无限重试。
- 匹配成功后不会留下会误取消订单的等待超时时间作业。
- 非作业人员工作时间不触发供给侧准入或投放。
- 页面入口可正常加载，提交前检查通过。

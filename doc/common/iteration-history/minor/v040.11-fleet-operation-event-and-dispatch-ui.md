# v040.11 Fleet Operation 事件反馈与任务调度边界

## 背景

本轮复核的问题集中在运维任务单闭环和策略边界：

- 清洁、充电、维修等运维任务的创建和操作仍使用弹出提示，未进入表单下方的最近任务事件。
- 最近任务事件存在全局消息混入，用户没有操作成功时也可能看到大量不相关事件。
- 运维策略点击一次后，策略执行和结果展示边界不清晰，容易表现为一次操作触发了很多执行。
- Robotaxi 管理页隐藏了清洁、充电、维修入口，破坏 Robotaxi 作为运维触发源的业务入口。
- 表格固定列宽导致字段头和内容难以对应，长字段无法查看完整内容。
- 任务优先级调度配置与任务调度策略形成两个入口，任务调度策略还暴露了不符合业务边界的人工执行按钮。

## 业务原则

- 运维策略是触发源：一次执行生成一条策略执行记录，并可产生多条策略结果和任务单。
- 任务调度策略是被调用策略：由 Robotaxi 直接触发、运维策略、Robotaxi 释放等业务动作调用，不作为人工源头执行。
- Robotaxi 管理是运维触发入口：前端始终提供触发入口，是否执行、排队或拒绝由服务层根据订单、任务和优先级判断。
- 业务结果必须进入单据事件：成功、失败、空结果和排队都写入可归属事件，不用弹出提示作为业务事实。

## 迭代内容

- 运维策略、Robotaxi 直接触发、目的地分配、路径规划、到达确认、作业开始、作业完成统一写入最近任务事件。
- 事件新增 `source_page` 归属字段，用于按当前任务表单过滤展示，不作为业务状态。
- 运维策略执行面板改为按当前策略过滤策略执行记录，保持“一次策略执行，多条执行结果”的结构。
- Robotaxi 管理恢复清洁、充电、维修、故障和退役触发入口，移除前端占用状态隐藏逻辑。
- 表格列宽改为依据字段头和当前数据内容估算，主表和事件表均使用自适应布局。
- 任务调度策略新增优先级、排队、中断和调用规则配置；旧任务优先级配置页隐藏，并由任务调度策略服务提供兼容配置。
- 移除任务调度策略人工“执行任务调度”入口和行操作，保留执行记录与结果页面用于查看被业务调用后的审计结果。

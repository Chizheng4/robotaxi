# v031.1 路径事实与行驶距离汇总小迭代方案

状态：已完成。

版本定位：小迭代。

核心目标：修复路径结果和行驶记录距离字段分叉问题，让 Route、运营行驶记录、履约行驶记录和服务订单各自只承担自己的事实职责。

---

## 1. 根因定位

1. Route 生成服务写入的是 `total_step_count` 和 `total_distance_m`，但页面和业务语义需要的是 Route 自身的 `route_step_count` 和 `total_distance_km`，导致模拟触发后路径结果字段不完整。
2. Trip 行驶推进只按“当前 Route”重算 `distance_traveled_km` 和 `distance_remaining_km`，没有把已完成的第一段路径累加到履约行驶记录本身，所以第一段和第二段会出现距离口径不一致。
3. RouteExecution 也只保存当前 route 的行驶距离，缺少“关联全部路径总距离”的汇总字段；未来异常重规划后会出现同类问题。
4. 服务订单的履约距离应是 Trip 的投影，不应独立计算第二套口径。
5. 详情页把当前位置、路径详情、路径历史和行驶进度混在窄栏中展示，用户看到的是堆叠信息，不是业务重点。

---

## 2. 底层事实职责

### Route：路径事实

Route 只记录路径本身，不记录行驶进度：

1. `route_step_count`：移动步数；
2. `total_distance_km`：路径总距离；
3. `total_distance_m`：兼容米制计算字段。

Route 不记录已行驶距离和剩余距离。

### RouteExecution / Trip：行驶事实

运营行驶记录和履约行驶记录都是 Robotaxi 行驶模拟对象，统一保存：

1. `total_distance_km`：关联所有路径的总距离；
2. `distance_traveled_km`：历史已完成路径距离 + 当前路径已行驶距离；
3. `distance_remaining_km`：当前路径剩余距离。

### ServiceOrder：履约距离投影

服务订单不重新计算距离，只投影关联 Trip：

1. `trip_total_distance_km`
2. `trip_distance_traveled_km`
3. `trip_distance_remaining_km`

---

## 3. 执行计划

1. 在路径规划服务中新增统一 route fact 与 travel metric 计算函数。
2. 所有路径生成函数统一写入 `route_step_count` 和 `total_distance_km`。
3. Trip 和 RouteExecution 的规划、步进、到达确认统一调用 travel metric 计算，支持多 Route 累计。
4. 服务订单同步 Trip 距离投影，不独立计算。
5. 优化 Trip 和 RouteExecution 详情结构：`行驶信息 / 位置信息 / 行驶进度 / 路径信息`，路径信息只展示关联路径摘要。
6. 同步字段字典和字段展示合同。
7. 新增自动验证覆盖 Route、Trip、RouteExecution 和 ServiceOrder 的距离事实。

---

## 4. 不纳入本轮

1. 异常自动模拟扩展；
2. 新指标体系；
3. 成本模型口径调整；
4. 地图算法替换。

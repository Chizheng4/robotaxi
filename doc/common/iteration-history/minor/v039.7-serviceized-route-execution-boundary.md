# v039.7 小迭代：运营行驶记录服务化边界与模拟隔离

## 目标

把用户确认的工程边界规则化、服务化、验证化：

1. 运营行驶记录是 Robotaxi 执行运营任务的通用行驶服务，不属于运营投放任务私有逻辑。
2. 清洁、充电、维修、故障处理、退役等 Fleet Operation 任务当前只构建业务基础闭环，暂不接入模拟运行主路径。
3. 模拟运行是上层自动化调用者，不能因为新增任务单而自动扫描或推进未声明接入合同的业务对象。
4. 其他模型后续迭代必须通过规则和脚本知道这些边界，不能靠用户每次重复提醒。

## 问题

- 现有 Fleet Operation 路径规划和到达确认仍位于 `src/main.jsx` 页面层。
- 运维任务生成的运营行驶记录可能复用投放任务的推进函数，而投放函数假设存在 `deploymentTask`。
- 模拟运行作用域对 `routeExecutions` 只按 `simulation_run_id` 和终态过滤，没有显式限制当前只扫描投放类运营行驶记录。
- 规则已有“业务服务复用”原则，但缺少“运营行驶记录通用服务”和“新增业务默认不接入模拟”的可执行表达。

## 执行计划

1. 更新 `AGENTS.md`、`05-codex-execution-rules.md`、`07-simulation-runtime-architecture-rules.md`，把业务服务边界写成强制规则。
2. 在 `fleetOperationTaskService` 中新增 Fleet Operation 路径规划、行驶推进和到达确认服务函数。
3. `src/main.jsx` 只调用服务函数，不再拼装 Fleet Operation 运营行驶记录。
4. `simulationRuntimeScope` 增加 routeExecution 参与模拟扫描的业务类型过滤，当前只允许 `DEPLOYMENT` 或历史兼容投放记录。
5. 新增 `verify-v039-7-route-execution-service-boundary.mjs` 并纳入提交前检查。
6. 重新生成 bundle，执行提交前检查和页面加载验证，完成版本记录、提交和 tag。

## 验收

- 规则能表达给其他模型：业务闭环服务化、页面不拼业务、运营行驶记录通用、模拟接入需显式合同。
- 运维任务路径规划生成的 routeExecution 带 `task_type`，但不会进入当前模拟工作流扫描。
- 投放 routeExecution 仍可进入当前模拟扫描。
- 提交前检查能拦截页面层运维行驶拼装回流。

## 执行结果

- 已完成规则固化：AGENTS、执行规则、模拟运行架构规则补齐业务服务边界。
- 已完成服务下沉：Fleet Operation 路径规划、行驶推进和到达确认进入 `fleetOperationTaskService`。
- 已完成模拟隔离：当前模拟运行只扫描投放类运营行驶记录，运维行驶记录默认隔离。
- 已完成页面分流：运营行驶记录操作按 `task_type` 区分投放与运维。
- 已完成验证：提交前检查和真实浏览器加载验证通过。

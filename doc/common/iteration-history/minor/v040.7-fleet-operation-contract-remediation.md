# v040.7 小迭代：Fleet Operations 架构合同止血修复

## 目标

修复 v039/v040 中“计划已声明但代码半接入”的问题，优先补齐字段展示合同、Fleet Operation 调度合同、Robotaxi 运维标记展示和任务优先级实际接入，避免后续迭代继续建立在不完整实现上。

## 范围

1. 修复 `operation_tags` 字段字典缺失，并让 Robotaxi 表格真实显示运维标记。
2. 补齐运维调度结果页，避免 `fleetOperationDispatchDecisions` 只有运行态和持久化、没有前端入口。
3. 调度失败也生成稳定 run / decision 编号，保证历史复盘和表格 key 可用。
4. 将 `NEAREST_AVAILABLE` 调度从“取第一个中心”收敛为按 Robotaxi 与运营中心 Cell 距离排序的可解释选择。
5. 将任务优先级服务接入 Fleet Operation 任务创建入口：立即执行、排队、拒绝三类决策真实影响任务创建。
6. 补齐退役二次确认动作入口，修复退役按钮调用不存在函数的问题。
7. 扩展验证脚本覆盖本轮合同，重新生成 bundle 并执行提交前检查。

## 执行结果

- `operation_tags` 已同步到文档版和代码版字段字典，并在统一表格渲染函数中显示清洁、充电、维修、故障 tag。
- 新增运维调度结果页面接入：菜单、表格配置、对象类型、主键、状态字段、详情选择和字段字典均已补齐。
- `fleetOperationDispatchService` 失败路径会生成稳定的调度执行编号和调度结果编号。
- `NEAREST_AVAILABLE` 调度按 Cell 曼哈顿距离选择最近具备能力的运营中心，调度结果记录距离。
- `fleetOperationTaskService` 创建任务前调用 `robotaxiTaskPriorityService`，占用时根据配置写入 `pending_task_queue`，不再创建虚假的等待任务单。
- Robotaxi 退役按钮恢复二次确认，并由服务层将退役车辆移出可调度状态。
- `verify-v037-2-fleet-operation-services.mjs`、`verify-v038-fleet-operation-policy.mjs`、`verify-v039-runtime-load-contract.mjs` 已扩展对应合同。

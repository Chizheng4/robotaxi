# v029.4 小迭代：人工与模拟业务执行同源化

## 背景

v029.3 修复了自动模拟不生成 Route、RoutePlanningRun 和行驶事实的问题，但实现方式仍然存在架构偏差：模拟侧新增了路径规划逻辑，容易与人工点击的原始业务闭环分叉。

本轮根据最新确认原则修正方向：

```text
人工点击功能本身是业务基础能力。
自动模拟不是重新创造功能，而是自动触发同一套业务能力。
Robotaxi 行驶必须基于已生成的 Route。
服务订单“已支付”只能是 payment_status，不是 order_status。
```

## 目标

1. 人工点击与自动模拟共享路径规划、行驶和支付的核心业务服务。
2. 履约行驶记录路径规划必须生成并引用 Route / RoutePlanningRun。
3. Robotaxi 行驶推进必须依赖 Route steps；没有 Route 不得计算位置、距离和剩余距离。
4. 服务订单待支付状态下执行支付，支付完成后订单主状态为已完成，支付状态为已支付。
5. 验证脚本要覆盖上述同源执行约束，防止后续回退。

## 执行内容

- `src/main.jsx`
  - 加载共享路径规划服务。
  - 人工价格预估路径、履约路径、运营投放路径和路径规划运行记录包装函数委托到共享服务。
  - 人工支付操作改为调用订单服务层支付函数。

- `src/services/routePlanningService.js`
  - 作为人工与模拟共同使用的路径规划服务来源。
  - 公开运营投放路径和履约路径创建函数。
  - 路径历史支持传入触发类型和创建时间，人工与自动可共用结构。

- `src/services/serviceOrderService.js`
  - 新增 `payServiceOrder`，统一封装服务订单支付闭环。
  - 明确订单主状态从待支付进入已完成，支付状态记录已支付。

- `src/services/simulationHandlers.js`
  - `PAYMENT_EXECUTE` 不再直接改状态，改为调用 `serviceOrderService.payServiceOrder`。

- `scripts/verify-simulation-strategy-execution.mjs`
  - 增加共享路径服务生成接驾路径的验证。
  - 增加支付后订单状态与支付状态分离验证。

- `scripts/verify-simulation-continuity.mjs`
  - 排空阶段支付验证注入真实订单服务和 Trip setter，保证模拟支付不绕服务执行。

## 后续原则

后续如继续收敛准入、匹配、结算、异常重规划等链路，应优先把人工点击逻辑服务化，再让模拟调用服务；不要再在模拟 handler 中复制一套业务功能。

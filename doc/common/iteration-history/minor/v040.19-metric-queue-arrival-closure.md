# v040.19 Metric Queue Arrival Closure

## 背景

本轮修复三类底层闭环问题：经营分析洞察卡片出现 `undefined`、Robotaxi 任务规划队列序号不唯一、运维任务与运营行驶记录的到达关系被过度收敛。硬边界是：运营行驶记录已经是既有服务化闭环，本轮只修正运维任务对运营行驶记录的调用和状态投影，不重写、不改造运营行驶记录服务。

## 已确认问题

1. 经营分析中财务结果、服务效率、过程质量有统计数据但显示 `undefined`。
2. Robotaxi 当前有任务时，后续排队任务可能都显示序号 1，队列无法表达明确执行顺序。
3. 任务规划综合状态只表达队列数量，缺少当前占用、当前任务优先级、队列明细和下一排队任务。
4. 任务完成后 Robotaxi 没有按排队序号自动激活下一任务。
5. 运维任务行驶到达被 v040.18 直接推进到 Worker 阶段，缺少“已到达目的地 -> 正常 / 异常到达确认”的统一行驶闭环。
6. “待分配作业人员”和“待分配 Worker”命名混用，状态筛选存在重复展示风险。

## 实现内容

1. 经营分析新增 `createMetricPresentation` 和 `createMetricInsightGroup`，洞察卡片只渲染标准展示对象，不再直接拼接可能缺失的指标字段。
2. `robotaxiTaskPriorityService` 新增 `normalizeQueuedTasks`，所有队列排序后统一重写唯一连续 `queue_sequence`。
3. `robotaxiTaskPlanningService` 增强综合状态，输出当前占用对象、当前任务优先级、标准化排队队列和下一排队任务。
4. 任务规划排队决策新增完整 `queue_snapshot`，任务规划执行和结果记录持久展示队列快照。
5. `fleetOperationTaskService` 将运维行驶到达投影恢复为 `ARRIVED_*`，正常到达确认后才进入待分配 Worker。
6. 新增 `confirmFleetOperationAbnormalArrival`，运维任务异常到达投影为 `ARRIVAL_ABNORMAL`。
7. 新增 `activateNextQueuedFleetOperationTask`，运维任务完成后按队列序号激活下一任务，并重排剩余队列。
8. 前端运维任务到达状态行操作继续查看运营行驶记录，待分配 Worker 才显示“分配 Worker”。
9. 字段字典同步新增 `queue_snapshot`、`current_assignment`、`current_task_priority`、`next_pending_task` 等字段，并统一待分配 Worker 文案。

# v040.12 Robotaxi 任务规划策略

## 背景

v040.11 修复了运维事件反馈和任务调度 UI 边界，但仍存在更底层的问题：同一 Robotaxi 是否能接收清洁、充电、维护、故障、投放或服务订单，仍由多个入口分散判断。这样会导致页面按钮、策略扫描、投放任务和订单匹配之间出现不一致。

本轮将问题收敛为业务底层闭环：任何业务对象想占用 Robotaxi，必须先经过 Robotaxi 综合状态和任务规划策略裁决。

## 业务模型

Robotaxi 状态拆为五层：

- 运营状态：可运营 / 不可运营。
- 生命周期阶段：运营预备阶段、运营阶段、退役阶段。
- 运营阶段：首次准入、准入修复、准入后待投放、持续运营、已退役。
- 车辆空间状态：停车中、临停中、行驶中等物理状态。
- 当前占用状态：无当前占用、运营准入任务、运营投放任务、服务订单、运维任务。

关键原则：

- 空闲不是车辆空间状态，而是“无当前执行任务 / 未被业务占用”的占用状态。
- 首次不可运营只能创建运营准入任务。
- 准入后但未进入运营的 Robotaxi 不应创建清洁、充电、维护任务。
- 持续运营阶段才允许按条件触发清洁、充电、维护。
- 故障处理主要来自行驶或运营异常上下文，不作为普通管理按钮随意触发。
- 退役是生命周期动作，不再作为普通 Robotaxi 运维按钮创建。
- Robotaxi 内部任务队列优先于外部服务订单和投放匹配。

## 实现内容

- 新增 `src/domain/robotaxiTaskPlanningTypes.js`，定义任务规划策略、综合状态和规划决策枚举。
- 新增 `src/services/robotaxiTaskPlanningService.js`，提供 `resolveRobotaxiCompositeState`、`planRobotaxiTask` 和 `getAvailableRobotaxiActions`。
- 运维任务创建服务 `fleetOperationTaskService` 改为先调用任务规划策略，再创建、排队或拒绝。
- 订单匹配引擎 `orderMatchingEngine` 使用任务规划策略筛选候选 Robotaxi。
- 自动/模拟投放入口 `businessActionService.createDeploymentTask` 使用任务规划策略筛选候选 Robotaxi。
- Robotaxi 管理页按钮由 `getAvailableRobotaxiActions` 预判显示，点击时仍由服务层实时裁决。
- 新增 `robotaxiTaskPlanningStrategies` 运行态集合、菜单页面、字段字典、快照保存/恢复和 bootstrap 等待链。
- 升级 v037/v038/v040.8/v040.9 历史验证合同，使其符合“持续运营阶段才允许普通运维、内部队列优先于外部分配”的新业务规则。

## 模拟边界

本轮不让模拟运行新增任何任务规划专用扫描路径。模拟运行只继续调度既有业务动作服务：

- 服务订单匹配成功或失败；
- 运营投放创建成功或失败；
- 无符合条件 Robotaxi；
- 业务动作产生的单据和事件。

任务规划策略属于业务底层服务，模拟运行只消费业务服务结果，不自己实现 Robotaxi 任务规划逻辑。

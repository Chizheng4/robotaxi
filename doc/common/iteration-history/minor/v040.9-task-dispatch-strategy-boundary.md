# v040.9 任务调度策略与 Robotaxi 分配边界

## 背景

v040.8 完成了运维任务单生命周期闭环，但把“存在排队运维任务 / 运维标记”扩散成订单匹配、运营投放和支付释放链路中的硬阻塞。这会让 Robotaxi 是否进入运营池由多个局部规则共同决定，容易影响模拟运行中服务订单、收入、成本和后续经营分析事实。

本次迭代收敛为：

- Robotaxi 是否可被订单匹配或投放候选读取，统一看 `availability_status`、`available_for_dispatch`、`current_order_id`、`current_task_id`。
- 排队运维任务是候选事实，不是天然阻塞事实。
- Robotaxi 从订单或任务释放后，下一步进入服务订单、投放任务还是排队运维任务，由“任务调度策略”产生执行和结果记录。
- 新增任务调度策略不接入模拟主扫描路径，只作为业务策略对象和人工执行入口存在，后续接入模拟需另行声明接入合同。

## 方案

### 1. 服务边界

- `robotaxiService.canAcceptServiceOrder` 与 `canAcceptDeploymentTask` 不再读取排队运维任务、清洁标记、低电量标记等隐式阻塞字段。
- `serviceOrderService.payServiceOrder` 只完成订单支付、释放当前订单占用，并在车辆未退役、未损坏、未被任务占用时恢复为可调度。
- `orderMatchingEngine` 与 `routePlanningService` 不再各自维护 Fleet Operation blocker，避免形成第二套候选规则。
- `fleetOperationTaskService` 在排队任务创建时只记录任务单和 Robotaxi 队列引用，不再把 Robotaxi 直接改为不可运营；真正占用 Robotaxi 的任务仍会设置 `current_task_id` 并变为不可调度。

### 2. 任务调度策略

新增三类对象：

- `TaskDispatchStrategy`：任务调度策略配置。
- `TaskDispatchRun`：策略执行记录。
- `TaskDispatchResult`：候选对象决策结果。

默认策略为“释放后任务调度”，使用 `RELEASED_ROBOTAXI_PRIORITY` 算法，按基础优先级和任务优先级补偿选择候选：

- 排队运维任务：基础优先级 80。
- 服务订单：基础优先级 60。
- 运营投放任务：基础优先级 40。

本次实现只生成策略执行事实和候选选择事实，不在页面层直接接管任务状态流转。

### 3. 前端与字段字典

- 运营中心管理新增：
  - 任务调度策略
  - 任务调度执行
  - 任务调度结果
- 策略页提供“执行任务调度”和行内“执行”操作。
- 两份字段字典同步新增任务调度字段、对象名和枚举中文。

## 验证

- 新增 `scripts/verify-v040-9-task-dispatch-boundary.mjs`。
- 验证排队运维任务不再把 Robotaxi 改为不可运营。
- 验证服务订单支付释放后 Robotaxi 恢复可调度。
- 验证订单匹配不因 `pending_fleet_task_id` 排除 Robotaxi。
- 验证任务调度策略能选中排队运维任务候选。

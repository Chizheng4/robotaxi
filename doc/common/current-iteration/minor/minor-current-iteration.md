# v022.6 服务订单结算闭环断点修复计划

## 当前问题

服务订单处于 `SETTLING`（结算中）状态时，点击“结算”后未完成闭环。

## 业务原则

1. 服务订单进入结算中，必须已经完成履约行驶。
2. 最终/实际行驶距离必须等于履约行驶记录中的“履约已行驶距离（公里）”，即 `Trip.distance_traveled_km`。
3. 最终费用计算策略不能用不符合业务语义的报价兜底绕过实际履约数据。
4. 最终费用计算策略的输入字段必须结构化、含义明确，并来自服务订单与履约行驶记录。

## 修复范围

1. 撤销不符合业务语义的 `quoted_price` 兜底结算逻辑。
2. 修复服务订单结算按钮闭环：
   - 服务订单结算中；
   - 调用最终费用计算策略；
   - 生成定价执行记录；
   - 生成最终定价结果；
   - 写回最终价格；
   - 服务订单进入待支付。
3. 严格整理最终费用计算输入字段：
   - 服务订单编号；
   - 履约行驶记录编号；
   - 实际/最终行驶距离；
   - 实际/最终行驶时长；
   - 报价起步价；
   - 报价距离单价；
   - 报价时间单价；
   - 报价动态系数；
   - 原始客户报价；
   - 必要的上下车位置、Robotaxi 和客户上下文。
4. 在服务订单管理表单下方增加“最近事件记录”。
5. 服务订单所有功能操作必须写入最近事件记录，包括：
   - 创建服务订单；
   - 价格预估；
   - 呼叫 Robotaxi；
   - 分配 Robotaxi 成功/失败；
   - 创建履约行驶；
   - 结算成功/失败；
   - 支付成功/失败。

## 验证方式

1. 代码级闭环验证优先：
   `SETTLING 服务订单 -> 结算 -> 定价执行 -> 定价结果 -> 待支付`
2. UI 只做关键冒烟验证：
   - 服务订单能显示待支付；
   - 最终价格可见；
   - 最近事件记录可见；
   - 结算事件结果可见。

# v022 自动执行控制计划

## 1. 目的

本文档用于固化 `v022` 阶段的大版本迭代计划。

`v022` 的目标是继续优化运营后台在订单匹配、服务订单、履约行驶记录和路径规划结果上的中文表达、状态口径、表单结构和闭环操作，让当前 demo 更接近真实 Robotaxi 运营人员的使用体验。

本阶段以节省 Codex 使用量为重要原则：先把本轮需求整理为明确的子版本计划，后续按计划连续执行；每个子版本只做必要 diff、必要验证和必要提交，避免重复分析、反复确认和无关文件改动。

## 2. 阶段基线

```text
阶段基线版本：v021.6
阶段名称：订单匹配、服务订单与履约行驶体验优化
当前执行模式：按自动计划连续执行，遇到冲突再暂停确认
当前计划起点：v022.1
当前计划终点：v022.5
```

## 3. 使用量节省原则

1. 每个子版本开始前只读取本子版本涉及文件，不做全仓反复扫描。
2. 优先复用已存在的表单模块、详情模块、事件区和字段字典能力。
3. 字段、状态、菜单名只在统一字典和配置入口改一次，再由前端统一显示。
4. 能通过白名单隐藏的历史兼容状态，不立即删除底层枚举，避免扩大迁移成本。
5. 每个子版本只做一次必要校验；涉及前端交互时再做一次浏览器核心路径验证。
6. 完成一个稳定子版本即更新 `VERSION.md`、提交并打 tag。
7. 最后一个子版本执行 Major 归档收口，不再另开补丁版本补归档。

## 4. 统一执行假设

1. `订单那匹配结果` 按笔误处理，统一为 `订单匹配结果`。
2. `服务履约记录` 面向前端统一改名为 `履约行驶记录`；底层对象仍为 `Trip`。
3. 履约行驶记录完成时机按真实服务闭环处理：客户下车后，履约行驶记录进入 `COMPLETED`；结算中和待支付属于服务订单状态。
4. 历史状态优先通过前端状态筛选白名单和状态迁移兼容处理，不直接删除底层枚举。
5. `Route 管理` 改为 `路径规划结果`，并归入 `运营决策中心 / 路径规划管理` 下，顺序在 `路径规划执行` 后。

## 5. 总体目标

### 5.1 运营准入任务单

1. 前端状态中文显示优化：
   - `WAITING_ASSIGNMENT`：待分配；
   - `WAITING_CHECK`：待检查。
2. 保持底层状态值不变，避免破坏已有任务数据。

### 5.2 订单匹配管理

1. 菜单和页面标题统一：
   - `订单匹配策略` 保持为策略对象；
   - `匹配执行管理` 改为 `订单匹配执行`；
   - `匹配决策管理` 改为 `订单匹配结果`。
2. `订单匹配结果` 表格和详情中显示 Robotaxi 当前位置。
3. Robotaxi 当前位置应使用系统已有结构化位置表达，避免只显示孤立 `current_cell_id`。

### 5.3 服务订单

1. 服务订单表格和详情中显示 Robotaxi 当前位置。
2. 服务订单不再优先展示孤立的：
   - 实际距离（公里）；
   - 实际时长（分钟）。
3. 服务订单应同步展示关联履约行驶记录的：
   - 总距离（公里）；
   - 已行驶距离（公里）；
   - 剩余距离（公里）。
4. 修复 `SETTLING` 状态下结算操作无效的问题。
5. 结算操作必须调用最终费用计算策略，并产生定价执行记录和最终价格结果。
6. 结算输入使用价格预估阶段已快照的起步价、时间单价、距离单价、综合系数等，以及履约行驶记录形成的实际总距离和实际总耗时。
7. 结算成功后：
   - 服务订单状态进入 `WAITING_PAYMENT`；
   - 前端显示为 `待支付`。
8. 待支付状态下操作按钮为 `立即支付`，支付成功后服务订单进入 `COMPLETED`。
9. 服务订单页面必须具备符合统一表单模块结构的最近事件 / 辅助信息区域。

### 5.4 履约行驶记录

1. 前端名称从 `服务履约记录` 改为 `履约行驶记录`。
2. 无效或历史状态不在前端筛选中展示：
   - 结算中；
   - 等待运营决策；
   - 异常失败；
   - 待开始；
   - 已分配；
   - 已到达上车点；
   - 乘客已上车。
3. 履约行驶记录的距离字段按所有关联 Route 汇总：
   - 总距离（公里）= 所有关联 Route 总距离之和；
   - 已行驶距离（公里）= 已完成历史 Route 距离 + 当前 Route 已行驶距离；
   - 剩余距离（公里）= 当前及未完成 Route 剩余距离。
4. 履约行驶记录绑定新 Route 后，距离展示不能只显示当前 Route，必须包含历史已行驶 Route。
5. 详情中的履约进度要基于当前状态、当前 Route 和历史 Route 正确表达。
6. 详情中的路径关联要同时展示当前路径和历史路径。
7. `履约执行事件` 改名为 `履约行驶事件`。
8. 履约行驶事件和运营行驶事件都要展示事件消息，且字段顺序为：
   - 事件类型；
   - 事件结果；
   - 事件消息；
   - 关联对象；
   - 创建时间 / 事件时间。

### 5.5 路径规划结果

1. `Route 管理` 前端名称改为 `路径规划结果`。
2. `路径规划结果` 从 `Robotaxi 管理` 移到：

```text
运营决策中心
  路径规划管理
    路径规划策略
    路径规划执行
    路径规划结果
```

3. 路径规划结果继续展示所有由路径规划策略生成的 Route。
4. 不改变 Route 的底层对象职责：Route 仍是路径规划结果，不是静态地图空间对象。

## 6. 子版本计划

### v022.1 文档计划、字段字典与命名口径

目标：先固定本轮命名、状态显示和字段口径。

交付：

1. 创建本文件并更新 `major-current-iteration.md` 指向本计划。
2. 更新统一字段字典：
   - 待分配；
   - 待检查；
   - 待支付；
   - 订单匹配执行；
   - 订单匹配结果；
   - 履约行驶记录；
   - 履约行驶事件；
   - 路径规划结果。
3. 补充必要文档说明：Trip 前端显示名为履约行驶记录，结算中属于 ServiceOrder，不属于 Trip。

验收：

1. 当前大版本入口指向 v022 自动执行计划。
2. 字段和值的中文命名一致。
3. 未改动无关业务对象。

### v022.2 菜单、标题与状态显示优化

目标：完成前端菜单、页面标题和状态中文显示的第一轮统一。

交付：

1. 运营准入任务单状态显示改为待分配、待检查。
2. 订单匹配菜单改为订单匹配执行、订单匹配结果。
3. 服务履约记录菜单和页面改为履约行驶记录。
4. Route 管理改为路径规划结果，并移动到路径规划管理下。
5. 服务订单待支付状态显示为待支付，按钮显示为立即支付。
6. 清理履约行驶记录前端状态筛选中的无效状态。

验收：

1. 菜单结构符合本计划。
2. 前端不出现本轮已明确废弃的旧显示文案。
3. 底层兼容状态未被误删。

### v022.3 服务订单结算与支付闭环修复

目标：修复服务订单结算中操作，并让最终费用计算结果真正回写订单。

交付：

1. 检查并修复 `SETTLING` 状态下结算按钮无效问题。
2. 结算调用最终费用计算策略。
3. 生成定价执行记录和最终价格结果。
4. 最终价格回写服务订单。
5. 订单进入待支付。
6. 立即支付后订单进入已完成。
7. 服务订单表格和详情显示关联履约行驶记录距离信息。

验收：

1. 结算操作可从服务订单页面完成。
2. 结算后能看到最终价格结果。
3. 支付后服务订单进入已完成。

### v022.4 履约行驶记录距离汇总、详情与事件区

目标：让履约行驶记录能正确表达当前 Route 与历史 Route 的整体行驶进度。

交付：

1. 履约行驶记录总距离、已行驶距离、剩余距离按所有关联 Route 汇总。
2. 详情中展示当前路径和历史路径。
3. 履约进度显示基于汇总距离和当前状态。
4. 履约行驶事件加入事件消息。
5. 运营行驶事件也调整事件消息字段顺序。
6. 服务订单页面补齐最近事件 / 辅助信息区域。

验收：

1. 重新规划或多段 Route 后，履约行驶记录距离不丢历史段。
2. 事件区字段顺序符合本计划。
3. 页面仍符合统一表单模块布局。

### v022.5 校验、版本记录与 Major 归档

目标：完成 v022 大版本收口。

交付：

1. 执行提交前检查。
2. 必要时做浏览器核心路径验证：
   - 菜单结构；
   - 服务订单结算；
   - 立即支付；
   - 履约行驶记录距离显示；
   - 路径规划结果位置。
3. 更新 `VERSION.md`。
4. 将本计划从当前目录归档到 `doc/common/iteration-history/major/`。
5. 将 `major-current-iteration.md` 恢复为暂无进行中的大版本计划。
6. 提交并打 tag。

验收：

1. 工作区干净。
2. tag 到 `v022.5`。
3. 当前 major 目录不遗留已完成计划。

## 7. 暂不处理范围

1. 不新增 Demand、Dispatch、Metric 等未在本轮明确要求的对象。
2. 不重写服务订单、Trip、Route 的底层对象模型。
3. 不删除历史兼容枚举。
4. 不调整真实地图或复杂调度算法。
5. 不改动与本轮菜单、状态、订单结算、履约距离无关的页面。

## 8. 风险与暂停条件

执行中如出现以下情况，需要暂停确认：

1. 发现“客户下车后完成”与现有文档存在明确冲突，且无法通过兼容方式处理。
2. 最终费用计算策略输入缺少必要价格快照或实际距离数据。
3. Route 历史记录无法可靠区分已完成段和当前段。
4. 服务订单事件区需要新增新对象或明显扩大任务系统范围。
5. 浏览器验证发现旧本地缓存数据会导致关键页面不可用。

# v032 统一时间世界与时间驱动执行引擎方案设计

状态：方案设计已建立，等待按自动化执行计划分版本推进。

版本定位：大版本架构迭代。

核心目标：把 Robotaxi 运营平台从“tick 自动点击工作流”升级为“统一时间源驱动业务作业”。人工处理、真实时间自动化和模拟时间自动化都必须调用同一套底层业务闭环函数，只允许时间源和触发方式不同，不允许出现平行模拟逻辑。

本版本暂不实现经营指标体系，不实现完整需求预测、供给计划和正式投放计划模型；这些能力需要等统一时间世界、行驶事实、供给释放和订单等待机制稳定后再进入指标建模。

---

## 1. 根本问题

当前系统已经完成了业务动作同源化、路径规划事实沉淀和距离事实统一，但仍存在四类根本问题。

### 1.1 行驶仍偏点击推进

运营行驶记录和履约行驶记录当前仍以“继续行驶 / 步进”作为主推进方式。虽然行驶步进已经依赖 Route，但真实运营中 Robotaxi 不应靠点击一步一步移动，而应该随着统一时间流动自然更新：

1. 已耗时；
2. 当前 Cell；
3. 当前路径步序号；
4. 已行驶距离；
5. 剩余距离；
6. 到达条件。

路径规划完成后，系统应根据 Route 移动步数、Cell 尺寸和 Robotaxi 默认速度计算预计到达时间。只有当统一时间达到预计到达时间后，才进入到达识别和正常 / 异常到达处理。

### 1.2 自动化仍像自动点击，不是真实时间作业

当前模拟 tick 会触发工作流动作，但动作多是“到 tick 就执行并直接完成”。这不足以模拟真实世界里的持续作业：

1. Worker 检查需要耗时；
2. Robotaxi 行驶需要耗时；
3. 到达识别需要耗时；
4. 订单匹配可以等待和重试；
5. 支付、结算、上车、下车也应具备耗时配置。

正确模型应是：业务动作开始后创建时间作业，作业在统一时间中执行，到计划完成时间后再调用底层业务闭环函数完成状态推进。

### 1.3 订单分配失败语义过早终态化

在统一时间推进下，如果 Robotaxi 正在接驾、履约、投放或准入检查，新订单当下找不到可用车辆是合理现象。这不应立即等同于订单终态失败。

一次匹配无车应表达为“本次匹配尝试未成功，订单继续等待分配”。只有超过等待 SLA、超过最大重试次数、客户取消或服务区无供给能力时，才进入真正失败或取消。

### 1.4 前端详情仍暴露内部历史结构

运营行驶记录和履约行驶记录的“路径信息”应面向用户展示关联路径摘要，而不是直接展示 `route_history` 或 `route_history_detail` 这种内部结构。

用户需要看到的是：

1. 当前关联路径；
2. 关联路径列表；
3. 起点和终点；
4. 移动步数；
5. 总距离；
6. 路径策略；
7. 规划结果；
8. 失败原因。

## 2. 统一时间世界

系统建立统一时间上下文 `TimeContext`。所有核心业务动作都从 `TimeContext` 获取当前时间，不在业务核心逻辑中随意直接读取 `new Date()`。

### 2.1 三种时间模式

| 时间模式              | 中文名      | 时间来源 | 触发方式        | 适用场景               |
| ----------------- | -------- | ---- | ----------- | ------------------ |
| `REAL_MANUAL`     | 真实时间人工处理 | 真实时间 | 人工点击        | 人工创建、人工处理、调试单据     |
| `REAL_AUTOMATION` | 真实时间自动运行 | 真实时间 | 自动调度器       | 人工创建单据后系统按真实时间自动运行 |
| `SIMULATION`      | 模拟时间自动运行 | 模拟时间 | 高速 tick 调度器 | 模拟运行对象驱动的运营仿真      |

三种模式共享同一套业务动作函数和状态机。差异只在：

1. 当前时间从哪里来；
2. 谁触发动作；
3. 时间推进速度是否等于真实世界。

### 2.2 TimeContext 建议字段

| 字段                       | 中文名     | 说明                                           |
| ------------------------ | ------- | -------------------------------------------- |
| `time_mode`              | 时间模式    | `REAL_MANUAL / REAL_AUTOMATION / SIMULATION` |
| `now`                    | 当前真实时间  | 真实 ISO 时间                                    |
| `simulation_run_id`      | 模拟运行编号  | 模拟模式下的来源运行对象                                 |
| `simulation_timeline_id` | 模拟时间轴编号 | 连续模拟运行所属时间轴                                  |
| `simulation_seconds`     | 当前模拟秒   | 从模拟时间轴开始累计的秒数                                |
| `simulation_timestamp`   | 当前模拟时间  | 例如 `Day 2 01:03:05`                          |
| `global_tick`            | 全局 Tick | 模拟模式下的全局 tick                                |
| `trigger_source`         | 触发来源    | 人工、真实自动化、模拟自动化                               |

### 2.3 时间记录原则

1. 真实创建时间、真实更新时间继续保存真实时间。
2. 模拟创建时间、模拟更新时间保存模拟时间。
3. 自动化作业必须记录计划开始时间、计划完成时间和实际完成时间。
4. 人工点击仍可立即完成动作，但也应通过同一 `TimeContext` 记录时间来源。

## 3. 时间驱动作业模型

新增统一概念：`TimedOperation`。

它不是某个单据本身，而是表示“一个业务动作正在随时间执行”。例如行驶、检查、到达识别、匹配重试等待、乘客上车、结算、支付等。

### 3.1 TimedOperation 建议字段

|字段|中文名|说明|
|---|---|---|
|`timed_operation_id`|时间作业编号|唯一编号|
|`time_mode`|时间模式|真实人工、真实自动化或模拟|
|`operation_type`|作业类型|行驶、检查、到达识别、匹配重试等|
|`operation_status`|作业状态|等待中、执行中、已完成、失败、取消|
|`object_type`|关联对象类型|Trip、RouteExecution、ReadinessTask、ServiceOrder 等|
|`object_id`|关联对象编号|业务对象编号|
|`action_type`|完成触发动作|到期后调用的底层业务动作|
|`started_at`|真实开始时间|真实时间|
|`planned_completed_at`|真实计划完成时间|真实时间模式使用|
|`completed_at`|真实完成时间|真实时间|
|`simulation_started_at`|模拟开始时间|模拟时间模式使用|
|`simulation_planned_completed_at`|模拟计划完成时间|模拟时间模式使用|
|`simulation_completed_at`|模拟完成时间|模拟时间|
|`duration_seconds`|计划耗时（秒）|配置或计算得到|
|`elapsed_seconds`|已耗时（秒）|由时间推进计算|
|`remaining_seconds`|剩余耗时（秒）|由时间推进计算|
|`progress_percent`|进度（%）|由时间推进计算|
|`failure_reason`|失败原因|失败时记录|

### 3.2 调度器职责

新增时间作业调度器 `TimedOperationScheduler`，每次时间推进时做四件事：

1. 读取当前 `TimeContext`；
2. 更新执行中作业的进度；
3. 找到已经到期的作业；
4. 调用底层业务动作服务完成作业，不直接手写业务状态。

调度器禁止直接改业务对象复杂状态，只能调用已有业务动作服务或明确新增的业务动作函数。

## 4. 行驶时间驱动模型

运营行驶记录 `RouteExecution` 和履约行驶记录 `Trip` 本质上都是 Robotaxi 的行驶模拟对象。它们应共享行驶进度计算模型。

### 4.1 路径规划完成后的动作

路径规划成功后：

1. 保存 Route、RoutePlanningRun 和路径摘要；
2. 创建 `TRAVEL` 类型时间作业；
3. 行驶对象进入“行驶中”；
4. 记录行驶开始时间和计划到达时间。

预计行驶时长：

```text
travel_duration_seconds = route_step_count * cell_travel_seconds
```

默认参数：

```text
Robotaxi 默认速度 = 30 km/h
Cell 默认长度 = 50 m
cell_travel_seconds = 6 秒
```

后续可以把速度、Cell 时长、到达识别时间迁移到配置功能。

### 4.2 行驶进度计算

随着统一时间推进：

```text
progress = elapsed_seconds / travel_duration_seconds
current_step_index = floor(route_step_count * progress)
distance_traveled_km = total_distance_km * progress
distance_remaining_km = total_distance_km - distance_traveled_km
time_elapsed = 历史已耗时 + 当前作业已耗时
```

如果行驶对象存在多条 Route：

```text
总距离 = 关联所有 Route 的总距离加总
已行驶距离 = 历史完成 Route 距离 + 当前 Route 已行驶距离
剩余距离 = 当前 Route 剩余距离
```

### 4.3 到达识别

当时间达到预计到达时间后，不应立即视为业务完成，而是进入到达识别作业：

1. 创建 `ARRIVAL_DETECTION` 时间作业；
2. 到达识别耗时由配置决定；
3. 第一阶段默认全部正常到达；
4. 未来可以按异常概率、道路阻塞、车辆故障等规则进入异常到达。

到达识别完成后调用底层业务动作：

1. 运营行驶记录：正常到达 / 异常到达；
2. 履约行驶记录：到达上车点 / 到达目的地；
3. 不允许调度器自己直接改最终状态。

## 5. 供给侧时间驱动

### 5.1 运营准入任务

准入任务不应只靠人工点击完成。自动化模式下：

```text
创建准入任务
-> 分配 Worker
-> 创建检查时间作业
-> 作业完成
-> 调用检查通过 / 检查异常业务动作
```

第一阶段默认检查正常通过，保留异常配置入口。

### 5.2 运营投放任务

投放任务在没有正式需求预测和供给计划前，使用临时供给再平衡模型，不再使用简单随机目标。

临时模型目标：

1. 避免车辆投放到当前位置；
2. 避免所有车辆集中到同一服务区；
3. 优先补充车辆密度低的服务区；
4. 兼顾距离成本；
5. 如果已有等待订单，优先考虑等待订单所在服务区；
6. 仍只从已定义服务区可用车辆点位中选择目标。

该模型是正式供给计划前的过渡能力，不等同于完整需求预测。

## 6. 需求侧时间驱动

### 6.1 服务订单匹配等待与重试

一次匹配没有可用 Robotaxi，不应立即终态失败。

建议流程：

```text
WAITING_ROBOTAXI_ASSIGNMENT
-> 匹配尝试
-> 无车：记录 MATCH_ATTEMPT_FAILED 事件，订单继续等待
-> 下一个重试时间到达后再次匹配
-> 超过 SLA 或最大重试次数：进入分配超时 / 分配失败
```

第一阶段可以少增状态，但必须区分：

1. 一次匹配尝试失败；
2. 订单仍在等待；
3. 订单终态失败。

### 6.2 乘客上车、下车、结算、支付

这些动作也应逐步由时间作业驱动：

1. 乘客上车耗时；
2. 乘客下车耗时；
3. 结算耗时；
4. 支付耗时。

第一阶段默认正常完成，未来再加入异常和概率规则。

## 7. 模拟运行对象语义

### 7.1 允许创建多个模拟运行

必须恢复多个模拟运行对象的创建能力。

原则：

1. 可以创建多个 `READY` 模拟运行对象；
2. 不允许多个 `RUNNING / DRAINING` 同时运行；
3. `PAUSED` 不阻止创建新运行，但启动新运行前必须处理 active run；
4. 新运行的起始模拟时间基于已有运行的最大结束时间或最大当前时间，保持时间连续。

### 7.2 FAILED 语义

`SimulationRun.FAILED` 不表示某个订单或某个业务动作失败。它只表示模拟运行对象自身无法完成，例如：

1. 时间作业排空超限；
2. 调度器无法继续推进；
3. 关键系统错误导致运行异常结束。

前端不得只显示“失败”而不提供定位。模拟运行详情必须展示：

1. 失败来源；
2. 失败动作；
3. 失败对象类型；
4. 失败对象编号；
5. 失败原因；
6. 最近失败事件。

建议显示文案：

|内部状态|前端文案|说明|
|---|---|---|
|`FAILED`|运行异常结束|运行对象自身异常终止|
|`DRAINING`|收尾执行中|计划 Tick 已结束，已触发自动化作业仍在完成|
|排空超限失败事件|收尾失败|作业排空超过限制|

## 8. 前端体验要求

### 8.1 路径信息详情

运营行驶记录和履约行驶记录详情中的“路径信息”必须改为用户友好的路径摘要。

展示结构：

1. 当前路径摘要：路径编号、策略、规划结果、起点、终点、移动步数、总距离；
2. 关联路径列表：按时间顺序展示每条 Route；
3. 失败路径展示失败原因；
4. 默认不展示内部 `route_history` 原始对象；
5. 如需展开，使用统一结构化详情组件按需展开。

### 8.2 时间作业展示

新增或接入时间作业展示能力：

1. 作业状态；
2. 已耗时；
3. 剩余时间；
4. 预计完成时间；
5. 进度；
6. 关联对象。

不做复杂仪表盘，不进入指标体系。

## 9. 工程原则

1. 一个时间源：所有业务动作通过 `TimeContext` 获取当前时间。
2. 一个调度器：自动化推进由时间作业调度器统一处理。
3. 一套业务闭环函数：人工、真实自动化、模拟自动化必须共享底层业务动作。
4. 状态机不分叉：时间作业不能创造对象外的新状态语义。
5. 字段字典同步：新增字段、状态、枚举必须同步文档版和代码版字段字典。
6. 验证合同优先：每个子版本必须补充低成本自动验证，防止后续模型漏改。
7. 渐进落地：先打时间基础，再改行驶，再改供需闭环，不一次性推翻全系统。
8. 节省使用量：每个子版本只读取和修改当前范围必需文件，不全量重构。

---

## 10. 不纳入 v032

1. 完整经营指标体系；
2. 完整需求预测模型；
3. 正式供给计划对象；
4. 复杂异常概率策略；
5. 外部真实地图、真实支付或真实车端系统；
6. 大规模视觉重设计。

v032 的目标是建立真实运营模拟的时间基础和执行基础，让后续成本、收入、指标和预测建立在可信业务事实之上。

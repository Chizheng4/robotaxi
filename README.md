# Robotaxi 城市运营模拟平台

> 将 Robotaxi 视为一个城市级实时动态供给系统，从最小运营单元出发，持续验证供需匹配、服务履约、车队保障与经营优化能否形成可运行、可观察、可复盘的闭环。

## 我的出发点

过去十多年，我经历过电商、物流和汽车等行业。复盘这些经历，我的优势不是掌握某个行业的固定答案，而是习惯从需求、供给、履约、组织、成本和系统之间的关系理解问题，并把复杂业务逐步转化为对象、规则、流程和指标。

我也清楚自己的边界：我不是自动驾驶技术专家，也还没有真实经营一座城市 Robotaxi 车队的经验。今天对这个行业的理解，更多来自既有专业经验的迁移、持续学习和系统化推演，仍然需要被真实业务不断检验和修正。

因此，这个项目不是为了包装一个已经成熟的答案，而是把我的思考、能力和不足都放进一个可以持续验证的过程里。

## 我的判断与兴趣

我越来越关注的，不只是某个环节如何优化，而是一个复杂供给系统如何被真正经营：如何围绕经营目标，把有限资源、波动需求、履约过程、组织协同和数据反馈连接起来，并在成本、效率、服务与风险之间做整体选择。

我将 Robotaxi 出行服务理解为一个**城市级实时动态供给系统**：

> 在一个有限区域内，如何用有限的、可移动的、受状态约束的车辆供给能力，持续匹配不确定、分散、波动的出行需求，并形成可运行、可反馈、可优化的经营闭环。

Robotaxi 集中了我长期感兴趣的问题：供需匹配、资产与产能、时空履约、调度、补能、异常、成本效率和组织协同。它也是一个足够复杂、足够长期、值得认真学习的新领域。

我希望从供应链专家继续向**系统型经营者**发展：不脱离专业去谈抽象管理，而是在专业基础上形成更完整的经营视角，把局部经验沉淀为可以执行、度量、复盘和复制的系统能力。

## 为什么做这个项目

我做这个项目，是想把判断变成实践，把兴趣变成持续投入，也把尚不确定的理解放到一个能够暴露问题的系统中。

它不是为了证明我会写代码，也不是为了假装已经懂得 Robotaxi 运营，而是为了：

- 把过去的供应链、履约、产品和组织经验迁移到新的业务场景；
- 把行业理解拆成真实的业务对象、生命周期、策略和经营指标；
- 用模拟运行观察结果，发现认知缺口，并持续修正；
- 从最小运营单元出发，探索单城市运营乃至城市复制所需要的能力。

我相信，真正有价值的专业成长，不是更熟练地重复已有经验，而是带着积累进入新的复杂问题，保持诚恳、耐心和行动，在验证中形成新的理解。

我期待这个项目能够持续进化：它可能不完美，也不会一次完成，但每一次迭代都应当让我更接近真实业务，更理解经营，也更清楚自己下一步需要学习什么。

## 项目定位

Robotaxi 不只是自动驾驶技术问题，也是一个复杂的运营经营问题。

自动驾驶能力决定车辆能否安全行驶；运营系统则需要进一步回答：有限且受状态约束的车辆，如何持续匹配不确定、分散和波动的出行需求，并在服务水平、运营成本、车辆效率与安全风险之间取得平衡。

本项目围绕这一问题，构建一个面向 Robotaxi 真实运营逻辑的虚拟经营执行系统。它从小区域、少量业务对象和可验证规则开始，把需求、供给、履约、保障、成本与经营指标连接起来，使运营过程能够被模拟、观察、度量和持续优化。

## 核心目标

项目当前聚焦“最小运营单元验证”，重点验证以下问题：

1. 需求是否能够被识别、生成并转化为服务订单；
2. 有限车辆能否在时间、空间、电量和运营状态约束下完成匹配；
3. 接驾、载客、结算、补能、清洁、维修和异常处理能否形成独立且可追溯的业务闭环；
4. 车辆利用率、空驶率、等待时间、完成率、成本和收入等经营结果能否被统一度量；
5. 不同运营策略能否通过模拟运行形成可比较、可复盘的结果；
6. 一次性的运营经验能否逐步沉淀为标准化、可复制的系统能力。

本项目不以复杂算法或视觉展示为起点，而以业务事实、单据生命周期和经营反馈是否成立作为判断基础。

## 核心经营闭环

```text
经营目标
  ↓
需求预测与供给规划
  ↓
Robotaxi 供给形成与运营准入
  ↓
需求生成、订单匹配与服务履约
  ↓
投放、补能、清洁、维修与异常保障
  ↓
成本、收入、效率与服务指标反馈
  ↓
策略调整与经营优化
```

系统遵循三项底层原则：

- **业务单据是事实来源**：订单、任务和行驶记录分别管理自己的创建、状态、动作、时间线、成本与结果。
- **车辆行为由业务驱动**：Robotaxi 是执行资源，不主动改变自身状态；投放、履约和保障动作必须通过业务服务发生。
- **模拟运行是上层扩展**：模拟系统只负责统一时间、调度到期动作和调用已有业务服务，不重新实现业务闭环。

## 当前系统能力

平台当前按企业经营闭环组织为八个能力域：

| 能力域 | 主要职责 |
| --- | --- |
| 空间与运营区域 | 描述地图、道路、地点、服务区域、运营中心和经营空间 |
| 客户与需求 | 管理客户、需求画像与模拟出行需求 |
| 供应管理 | 完成需求预测、供给规划、车辆形成、区域交付和运营准入 |
| 车队资产管理 | 管理 Robotaxi 生命周期、位置、电量、路径和行驶事实 |
| 车队运营管理 | 完成车辆投放、服务履约以及充电、清洁、维修、故障和退役保障 |
| 运营组织管理 | 描述 Worker 与运营组织执行能力 |
| 战略与经营指标 | 管理经营目标、策略、成本、收入和运营结果分析 |
| 自动运营模拟 | 提供统一模拟时间、工作流调度和策略执行环境 |

平台提供桌面与手机端访问能力，包含运营中控台、业务列表、筛选与搜索、对象详情、状态时间线、最近事件和版本更新记录。每位访问者的模拟数据保存在自己的浏览器中，彼此独立。

## 项目希望沉淀什么

最终希望沉淀的，不是某个页面、单一策略或一组静态文档，而是三类能够持续积累的成果：

- **行业理解**：形成对 Robotaxi 供需、履约、资产、组织、成本和安全关系的系统认识；
- **经营方法**：形成从目标到决策、执行、业务事实、数据反馈和优化的完整思考框架；
- **系统能力**：把经营方法固化为业务对象、生命周期、服务、策略、指标和模拟实验环境。

长期来看，希望这套能力能够从小区域验证逐步走向单城市优化、城市复制和多城市经营，并持续回答一个核心问题：复杂的供给系统，如何从依赖个人经验走向可解释、可验证和可复制的经营能力。

## 当前范围与边界

### 当前聚焦

- 虚拟运营区域与空间对象；
- 模拟客户需求与服务订单；
- Robotaxi 资产、位置、电量和运营状态；
- 订单匹配、接驾、载客、结算和收入事实；
- 车辆投放、充电、清洁、维修、故障处理和退役任务；
- 经营目标、供需策略、成本效率与服务指标；
- 统一时间下的自动运营模拟与结果复盘。

### 暂不包含

- 真实地图和真实道路网络接入；
- 自动驾驶感知、预测、决策与控制仿真；
- 强化学习等复杂调度算法；
- 全城市真实交通流仿真；
- 真实账号、共享数据库与多人协同编辑；
- 多城市资源配置与整体经营优化。

## 系统结构

```text
Robotaxi 经营执行系统
  ├─ 空间与运营区域
  ├─ 客户与需求
  ├─ 供应管理
  ├─ 车队资产管理
  ├─ 车队运营管理
  ├─ 运营组织管理
  ├─ 战略与经营指标
  └─ 自动运营模拟
```

项目采用前端单页架构：React 负责业务界面与本地运行态，Python 静态服务负责本地访问，GitHub Pages 用于公开演示。当前业务数据存储在浏览器本地，不依赖后端数据库。

进一步了解系统边界和业务闭环，请阅读 [系统总览](doc/00-system-overview.md)。

## 快速开始

### 环境要求

- macOS；
- Node.js；
- Python 3；
- Chrome 或 Chromium 浏览器。

### 本地运行

双击项目根目录中的 `start-robotaxi.command`。

启动程序会自动完成前端构建、静态服务启动和浏览器加载验证，然后打开：

```text
http://127.0.0.1:4173/
```

保持启动窗口开启即可持续访问；按 `Ctrl-C` 停止服务。

### 在线体验

公开演示地址：<https://chizheng4.github.io/robotaxi/>

网站支持桌面浏览器、手机浏览器和常见内嵌 WebView 的基础响应式访问。业务数据保存在访问者自己的浏览器中，不会与其他访客共享。

### 发布最新版本

Codex 完成版本验证、提交和标签后，双击项目根目录中的 `publish-robotaxi.command`。发布程序会自动：

1. 校验当前分支、工作区、版本记录、提交说明和版本标签；
2. 运行完整提交前检查；
3. 推送 `main` 与版本标签；
4. 等待 GitHub Actions 构建完成；
5. 校验公网部署清单已经切换到当前版本和提交。

只有线上网站确认是最新版时，发布程序才会提示成功。该命令完全在本机和 GitHub 中运行，不调用 Codex，也不消耗 Codex 使用量。

### 项目检查

双击 `check-project.command` 可运行项目提交前检查。

## 长期方向

项目将沿着以下层级逐步演进：

1. 最小运营单元验证；
2. 区域内持续优化与运营工程化；
3. 多区域扩展与区域间协同；
4. 单城市完整运营闭环；
5. 城市复制标准化；
6. 单城市持续优化与多城市整体经营优化。

每一阶段都以前一阶段的业务事实和可验证结果为基础，不用规模或复杂度替代闭环质量。

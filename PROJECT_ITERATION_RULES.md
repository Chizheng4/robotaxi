# Robotaxi 项目迭代规则

本文档用于约束 Robotaxi 模拟运营系统的后续迭代方式，保证文档、字段、代码、校验、前端展示和版本记录形成稳定闭环。

## 1. 迭代基本原则

- 每一轮迭代必须先明确当前目标，不一次性实现未来对象。
- 当前阶段只实现用户确认过的业务对象、初始化数据、校验规则和前端展示。
- 发现业务定义冲突、字段重复、对象边界不清时，必须先指出并确认，不直接编码。
- 保持对象原子化，避免为了当前页面把业务关系写死。
- 不重写无关文件，不删除旧版 demo，不破坏已有版本回退能力。

## 2. 标准迭代闭环

每次重要迭代默认按以下顺序执行：

1. 查看 Git 当前版本和工作区状态。
2. 阅读本轮指定文档。
3. 输出业务理解、差异分析、风险和计划修改文件。
4. 等用户确认后再修改文档或代码。
5. 如涉及业务对象、字段、枚举或状态，必须同步字段字典。
6. 编码只覆盖本轮确认范围。
7. 执行初始化校验、语法检查、前端可视化检查。
8. 涉及前端交互时，必须做本轮核心路径的页面模拟测试，确认菜单、按钮、表格、详情和状态变化符合预期。
9. 用户前端确认后再提交版本。
10. 提交后更新版本记录。

## 3. 文档规则

- 业务对象定义必须在文档中表达清楚，包括对象含义、字段、状态、枚举、约束和关系。
- 文档中新增对象、结构或逻辑时，必须检查是否影响初始化文件。
- 初始化文档应突出生成顺序、数据结构、约束和校验规则，避免重复长篇解释业务对象。
- 文档与代码不一致时，优先先修正文档冲突，再进入编码。
- 运营人员最终看到的表达以中文为主，文档中应保留英文字段名和中文显示名的对应关系。

## 4. 字段字典规则

- 每次方案设计涉及新增或修改业务对象、字段、状态、枚举、功能逻辑时，默认维护字段字典。
- 同一业务含义只能有一个稳定字段名，避免多个字段表达同一含义。
- 字段字典必须包含英文字段名、中文显示名、字段含义，必要时补充取值范围。
- 前端表格、详情栏、校验结果应优先使用字段字典显示中文。
- 字段字典是文档、代码、前端显示之间的统一口径。

## 5. 编码规则

- 优先使用最小可运行代码。
- 保持数据初始化、业务类型、校验规则、前端展示分层清晰。
- 不提前实现 Vehicle、Demand、Order、DispatchTask、Trip、Metric 等未确认对象。
- 已有业务对象如 Map、Cell、Road、Place、ServiceArea、Zone、Route、OpsCenter、Robotaxi、Worker 应保持定义稳定，除非文档逻辑要求调整。
- 新增对象时，应同时考虑对象类型、初始化数据、校验规则、菜单入口、表格显示和详情展示。

## 6. 校验规则

- 初始化数据必须具备可执行校验。
- 校验规则应覆盖数量、唯一性、引用关系、空间约束、状态初始值和关键业务约束。
- 校验结果必须能在前端以中文展示。
- 如果文档要求发生变化，旧校验规则必须同步删除或改写，避免误导。

## 7. 前端体验规则

- 前端目标是运营平台，不是营销页。
- B 端运营系统以有效业务空间为核心，表格、中控台、地图、详情、功能内容应获得最大可用面积。
- 导航、边栏、标题和装饰区域应保持克制，在可读和可操作前提下尽量少占空间。
- 页面布局应偏高信息密度，避免大卡片、大留白、大标题和营销式视觉结构。
- 区域分隔优先使用细线、浅背景和紧凑间距，不依赖大面积空白制造层次。
- 运营人员看到的菜单、字段、状态、枚举、详情信息应以中文为主。
- 菜单应支持层级、展开/收缩、当前选中态和父级高亮。
- 对象列表应清晰展示核心字段，点击记录后右侧展示详情。
- 前端功能完成后应模拟真实运营人员操作路径，至少覆盖本轮新增或修改的菜单进入、按钮点击、记录选择、详情展示和关键状态变化。
- 地图空间可视化应保持清晰、克制、可解释，不引入真实地图 SDK，除非用户明确要求。
- 不同业务语义应使用可区分但不过度花哨的颜色。

### 7.1 字体与空间密度

- 页面空间优先，标题、说明、工具栏和装饰性区域不得占用过多业务内容空间。
- 字体大小必须分层统一，不允许各模块自行放大造成界面突兀。
- 默认字体层级：
  - `12px`：表格内容、详情字段、筛选项、说明文字、状态值、输入内容。
  - `13px`：一级 / 二级 / 三级菜单、按钮文字、重要字段标签。
  - `14px`：模块标题、详情标题、表格区标题。
  - `16px`：产品级标题上限，仅用于全局品牌或极少数全局页面标题。
- 普通业务页面禁止使用 `18px+` 大标题；新增页面应向菜单尺度靠拢。
- 新增或修改前端页面时，必须检查字体层级、标题占用、表格密度和详情区密度是否符合规则。

### 7.2 菜单内业务页面布局

- 运营中控台可以采用地图 + 详情的特殊布局；其他业务菜单默认采用紧凑型业务列表页布局。
- 业务列表页应优先按以下区域组织：
  - 页面标题区：显示模块名称和简短说明，保持紧凑。
  - 状态分类区：按对象或单据的核心状态提供 `全部 + 各状态` 快捷分类。
  - 搜索过滤区：按关键字段筛选记录，支持后续字段扩展。
  - 全局功能区：放对整个列表或筛选结果生效的操作，如生成任务、自动检查、批量操作、刷新、导出。
  - 表格内容区：展示业务记录，字段名和值优先中文。
  - 单行固定操作列：表格最右侧固定显示单条记录操作，横向滚动时仍可见。
  - 辅助信息区：展示事件、日志、校验结果等，但不得挤压主表格。
  - 底部信息区：展示图例、分页、筛选结果、统计数据或模块说明等低干扰辅助信息。
- 搜索过滤区必须形成闭环：筛选字段 + `查询` + `重置`。输入或选择条件后，默认通过 `查询` 应用筛选，`重置` 清空条件并恢复列表。
- 状态分类区是业务对象和单据页面的默认框架能力；后续新增对象或单据时，如存在状态字段，必须优先提供状态分类。
- 状态分类区与搜索过滤区必须联动：点击 `全部` 或某个状态分类时，应以状态分类为主重新计算列表，并清空关键词、触发方式等临时筛选，避免跨模块跳转带来的筛选条件继续锁定数据。
- 全局操作和单行操作必须分开：全局操作放功能区，单条记录操作放表格右侧固定列。
- 单行操作按钮必须统一样式和结构，不允许在不同表格中混用随意按钮、纯文本状态或未结构化操作。
- 单行操作列只显示当前记录在当前状态下可执行的动作；不得用状态词冒充操作。
- 当前记录无状态变更动作但可查看对象时，统一显示 `查看详情`。
- `查看详情` 应选中当前记录；如果右侧详情栏隐藏，则同时展开详情栏。
- 跨模块关联操作应完成完整跳转闭环，例如 `查看行驶记录` 应切换到对应菜单、按关联任务或对象筛选、选中目标记录并展开详情。
- 底部信息区应保持紧凑，使用细线与主内容分隔；中控台可展示地图图例，业务列表页可展示记录数、筛选状态、分页或统计信息。
- 右侧详情栏必须跟随当前菜单和当前选中记录，不允许显示无关对象详情。
- 右侧详情栏应可折叠 / 展开，避免长期占用业务主区域空间。
- 右侧详情栏中的结构化位置上下文应默认展示摘要，例如网格、服务区、地点、道路片段等核心信息；完整 CellContext 可折叠展开，不应默认铺满冗长文本。
- 单据操作影响业务对象状态时，必须同步更新相关对象，并通过校验规则验证对象状态与单据状态一致；例如准入任务分配 Worker 后，Robotaxi 必须进入运维检查中。
- 前端模拟运行态应具备最小持久化能力；刷新页面不得丢失已生成的任务、对象状态、事件日志和用户当前页面状态，除非用户明确执行重置模拟数据。
- 菜单页面状态必须按页面维度保存；首次进入页面默认展示全部数据，非首次返回页面时可恢复该页面上次筛选、分类和选中记录。

## 8. 版本规则

- 每次稳定迭代完成后，应提交一个自增版本。
- 版本号按 `v001`、`v002`、`v003` 顺序递增。
- 每次提交后必须更新 `VERSION.md`，用中文简洁记录该版本核心变化。
- 版本记录应突出“做了什么”和“为什么重要”，避免写成冗长流水账。
- 提交前必须确认工作区变更范围，不把无关文件混入版本。

## 9. 风险处理规则

- 如果用户在编码过程中修改了文档或文件，应重新检查变更影响。
- 如果发现文档和代码冲突，应先说明冲突和建议处理方式。
- 如果某个迭代会扩大范围，应明确拆分为后续版本，不在当前版本偷跑。
- 如果前端展示出现英文枚举或误导性字段，应优先补字典或详情逻辑。
- 如果 Git 工作区不干净，应先判断哪些变更属于本轮，避免覆盖用户修改。

## 10. 当前默认记忆规则

- 方案设计更新后，默认检查并维护字段字典。
- 每次版本提交后，默认更新 `VERSION.md`。
- 用户确认前，不主动提交版本。
- 用户要求进入编码前，先确认当前版本基线和计划修改范围。

# Robotaxi 项目迭代规则

本文档用于约束 Robotaxi 模拟运营系统的后续迭代方式，保证文档、字段、代码、校验、前端展示和版本记录形成稳定闭环。

## 1. 迭代基本原则

- 每一轮迭代必须先明确当前目标，不一次性实现未来对象。
- 当前阶段只实现用户确认过的业务对象、初始化数据、校验规则和前端展示。
- 发现业务定义冲突、字段重复、对象边界不清时，必须先指出并确认，不直接编码。
- 保持对象原子化，避免为了当前页面把业务关系写死。
- 不重写无关文件，不删除旧版 demo，不破坏已有版本回退能力。

## 2. 标准迭代闭环

每次重要迭代默认按以下顺序执行：

1. 查看 Git 当前版本和工作区状态。
2. 阅读本轮指定文档。
3. 输出业务理解、差异分析、风险和计划修改文件。
4. 等用户确认后再修改文档或代码。
5. 如涉及业务对象、字段、枚举或状态，必须同步字段字典。
6. 编码只覆盖本轮确认范围。
7. 执行初始化校验、语法检查、前端可视化检查。
8. 用户前端确认后再提交版本。
9. 提交后更新版本记录。

## 3. 文档规则

- 业务对象定义必须在文档中表达清楚，包括对象含义、字段、状态、枚举、约束和关系。
- 文档中新增对象、结构或逻辑时，必须检查是否影响初始化文件。
- 初始化文档应突出生成顺序、数据结构、约束和校验规则，避免重复长篇解释业务对象。
- 文档与代码不一致时，优先先修正文档冲突，再进入编码。
- 运营人员最终看到的表达以中文为主，文档中应保留英文字段名和中文显示名的对应关系。

## 4. 字段字典规则

- 每次方案设计涉及新增或修改业务对象、字段、状态、枚举、功能逻辑时，默认维护字段字典。
- 同一业务含义只能有一个稳定字段名，避免多个字段表达同一含义。
- 字段字典必须包含英文字段名、中文显示名、字段含义，必要时补充取值范围。
- 前端表格、详情栏、校验结果应优先使用字段字典显示中文。
- 字段字典是文档、代码、前端显示之间的统一口径。

## 5. 编码规则

- 优先使用最小可运行代码。
- 保持数据初始化、业务类型、校验规则、前端展示分层清晰。
- 不提前实现 Vehicle、Demand、Order、DispatchTask、Trip、Metric 等未确认对象。
- 已有业务对象如 Map、Cell、Road、Place、ServiceArea、Zone、Route、OpsCenter、Robotaxi、Worker 应保持定义稳定，除非文档逻辑要求调整。
- 新增对象时，应同时考虑对象类型、初始化数据、校验规则、菜单入口、表格显示和详情展示。

## 6. 校验规则

- 初始化数据必须具备可执行校验。
- 校验规则应覆盖数量、唯一性、引用关系、空间约束、状态初始值和关键业务约束。
- 校验结果必须能在前端以中文展示。
- 如果文档要求发生变化，旧校验规则必须同步删除或改写，避免误导。

## 7. 前端体验规则

- 前端目标是运营平台，不是营销页。
- 运营人员看到的菜单、字段、状态、枚举、详情信息应以中文为主。
- 菜单应支持层级、展开/收缩、当前选中态和父级高亮。
- 对象列表应清晰展示核心字段，点击记录后右侧展示详情。
- 地图空间可视化应保持清晰、克制、可解释，不引入真实地图 SDK，除非用户明确要求。
- 不同业务语义应使用可区分但不过度花哨的颜色。

## 8. 版本规则

- 每次稳定迭代完成后，应提交一个自增版本。
- 版本号按 `v001`、`v002`、`v003` 顺序递增。
- 每次提交后必须更新 `VERSION.md`，用中文简洁记录该版本核心变化。
- 版本记录应突出“做了什么”和“为什么重要”，避免写成冗长流水账。
- 提交前必须确认工作区变更范围，不把无关文件混入版本。

## 9. 风险处理规则

- 如果用户在编码过程中修改了文档或文件，应重新检查变更影响。
- 如果发现文档和代码冲突，应先说明冲突和建议处理方式。
- 如果某个迭代会扩大范围，应明确拆分为后续版本，不在当前版本偷跑。
- 如果前端展示出现英文枚举或误导性字段，应优先补字典或详情逻辑。
- 如果 Git 工作区不干净，应先判断哪些变更属于本轮，避免覆盖用户修改。

## 10. 当前默认记忆规则

- 方案设计更新后，默认检查并维护字段字典。
- 每次版本提交后，默认更新 `VERSION.md`。
- 用户确认前，不主动提交版本。
- 用户要求进入编码前，先确认当前版本基线和计划修改范围。
